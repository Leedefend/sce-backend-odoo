# Native Capability Intent Runtime Exposure Spec v1

## 1. Objective

Freeze phase-4 platform contract for native capability runtime exposure so
native projected capabilities and platform-defined capabilities can share one
intent/runtime surface under `app_config_engine` capability core.

## 2. Layer Boundary

- Native adapters own discovery/projection only.
- Capability registry owns normalized capability assets only.
- Intent runtime owns execution dispatch only.
- Frontend exposure owns rendering only.

No layer may absorb another layer's ownership.

## 3. Scope

This spec applies to native projected capability types:

- `native_menu_entry`
- `native_window_action`
- `native_model_access`
- `native_server_action`
- `native_report_action`
- `native_view_binding`

## 4. Runtime Binding Contract

Each native capability row must expose an explicit binding payload.

### 4.1 Canonical Shape

```json
{
  "identity": {
    "key": "native.*",
    "type": "native_*"
  },
  "binding": {
    "intent": {
      "primary_intent": "...",
      "secondary_intents": []
    },
    "contract": {
      "subject": "...",
      "contract_type": "...",
      "contract_version": "v1"
    },
    "exposure": {
      "entry_mode": "menu|action|model|server_action|report|view_binding",
      "entry_ref": "xmlid|id"
    }
  },
  "permission": {
    "access_mode": "execute|readonly|locked|preview"
  },
  "runtime": {
    "supports_entry": true,
    "supports_execute": false,
    "safe_fallback": "workspace.home"
  }
}
```

### 4.2 Binding Invariants

- executable entry types must provide `binding.intent.primary_intent`
- contract-driven types must provide `binding.contract.subject`
- entry-capable types must provide `binding.exposure.entry_mode`
- runtime fallback must be explicit for entry types

## 5. Type-to-Intent Baseline

- `native_menu_entry` -> `app.open` (primary)
- `native_window_action` -> `ui.contract` (primary)
- `native_model_access` -> `ui.contract` (primary, readonly default)
- `native_server_action` -> `execute_button` or governed equivalent
- `native_report_action` -> report runtime intent
- `native_view_binding` -> `ui.contract` (binding projection, non-entry)

If runtime intent aliases are introduced, they must be declared in binding and
validated by registry lint before release.

## 6. Runtime Exposure Channels

Projection targets must be generated by platform projection services only:

- capability list projection
- capability matrix projection
- workspace exposure projection
- governance exposure projection

Frontend must consume projection output and must not infer native intent mapping.

## 7. Governance Guards

Phase-4 requires all guards below to remain PASS:

- `verify.architecture.native_capability_ingestion_guard`
- `verify.architecture.native_capability_ingestion_lint_bundle`
- `verify.architecture.native_capability_projection_coverage_report`
- `verify.architecture.native_capability_projection_snapshot_guard`
- `verify.architecture.native_capability_projection_release_guard_bundle`

## 8. Stop Rules

Stop and split into screen/verify batch when any condition appears:

- runtime intent cannot be resolved without adding domain-specific heuristics
- exposure needs frontend model-specific branching
- binding requires scene internals not supplied by platform interface
- native projection introduces duplicate ownership against platform assets

## 9. Acceptance Checklist

- six native types retain explicit binding payloads
- release guard bundle passes without baseline drift
- controller boundary guard remains PASS
- no industry module becomes runtime intent owner for native projection
