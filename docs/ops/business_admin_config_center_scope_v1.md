# 业务管理员可视化配置中心 v1 · Phase 1 配置化对象盘点

## 1. 目标与边界

本阶段目标：在不改业务代码的前提下，盘点并分级可配置对象，形成后续 Phase 2~4 的输入基线。

硬边界：
- 不改变业务事实真相（模型语义、权限真相、流程真相不在本阶段变更）。
- 不破坏 `contract freeze surface`（已冻结 contract 字段面不在本阶段调整）。
- 不引入通用低代码引擎（仅做业务管理员最小可用配置能力）。

## 2. 盘点范围

基于当前已成立的业务对象主线：
- `project.project`
- `project.task`
- `project.budget`
- `project.cost.ledger`
- `payment.request`
- `sc.settlement.order`

配置中心仅覆盖三类治理面：
- 系统参数/配置项
- 业务角色入口配置
- 首页/区块展示配置

## 3. 配置化对象分级

## 3.1 立即配置化（v1 首批）

1) 系统参数配置（后端参数层）
- 配置项分组：运行时开关、默认值、显示策略、提示文案。
- 约束：仅允许白名单参数；禁止写入会改变业务事实判断的关键字段。

2) 业务角色入口配置（导航层）
- 配置对象：按角色显示入口、入口顺序、入口可见性（显隐）。
- 约束：入口显隐不等于权限放行；真实访问结果仍由后端权限/规则决定。

3) 首页/区块配置（展示层）
- 配置对象：首页区块启用状态、排序、分组、默认折叠状态。
- 约束：仅改可发现性与呈现顺序，不改区块数据来源语义。

## 3.2 下一步配置化（v1.1/v2）

1) 角色模板与入口包
- 按业务岗位提供可复用入口模板（例如 PM/财务/业务负责人）。
- 保留人工覆写能力，但不越权。

2) 对象级默认表单偏好
- 仅配置默认 tabs/section 展开状态、默认列表筛选保存。
- 不允许配置写入后端规则语义。

3) Contract 输出中的“配置衍生字段”
- 仅新增 non-breaking、可选字段作为 UI 提示。
- 必须通过独立契约验收批次后进入冻结面。

## 3.3 暂不配置化（本轮排除）

1) 权限规则与 record rule 逻辑
- 原因：属于业务真相层，不应由可视化配置直接改写。

2) 审批流引擎与流程编排 DSL
- 原因：复杂度高，超出“业务管理员最小配置能力”范围。

3) 通用页面拖拽搭建器/低代码平台
- 原因：与本期“壳层配置中心”目标不一致，风险与成本过高。

## 4. Phase 2~4 设计输入（预案）

## 4.1 Phase 2：后端配置模型（最小治理模型）
- 建议最小模型：
  - `config.item`（系统参数白名单项）
  - `config.role.entry`（角色入口配置）
  - `config.home.block`（首页区块配置）
- 治理要求：审计字段完整（创建人、更新时间、版本、启停状态）。

## 4.2 Phase 3：前端配置中心壳层
- 页面形态：列表 + 表单，不做拖拽引擎。
- 交互目标：可搜索、可编辑、可预览变更影响范围。

## 4.3 Phase 4：配置结果与契约联动
- 进入 contract 的范围：仅导航/展示层可选字段。
- 仅作用后端参数的范围：系统默认值与开关。
- 联动原则：先后端供给，再前端消费；禁止前端先假设配置语义。

## 5. 风险清单

- 风险 1：把“入口可见”误解为“权限可访问”。
- 风险 2：配置项越界影响业务事实语义。
- 风险 3：未建立白名单导致配置漂移。

缓解策略：
- 所有配置对象按白名单管理。
- 配置生效前执行最小验证脚本（参数合法性 + 入口可达性 + deny-path 保持）。

## 6. 本阶段结论

- `PASS`（盘点阶段）：配置化对象与边界已分级明确。
- 下一步进入 Phase 2 时，只实现“最小治理模型”，不扩展低代码能力。
