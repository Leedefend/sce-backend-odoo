# 迭代执行规则 v1

## 1. 节奏上限规则

每个专题最多执行 3 批：

- Batch 1：建立最小可运行能力或证据
- Batch 2：收敛差异并补齐关键证据
- Batch 3：封板收口

超过 3 批默认视为节奏失控，需停止并重新立项。

## 2. 收口规则

当 `acceptance-pack PASS`：

- 必须立即 `STOP` 当前专题迭代
- 必须标记专题 `CLOSED`

不得继续“顺手优化”或“再补一轮验证”。

## 3. 验证占比规则

- verify 不得超过功能代码投入
- 禁止出现“主要工作都在写 verify、功能几乎不动”的开发形态

治理文档专题可不执行 verify，但必须显式记录“跳过依据”。

## 4. 方向规则

不允许 verify 驱动开发。

允许模式：

- 用户价值目标驱动开发
- verify 仅用于验收与守卫

禁止模式：

- 为了跑通 verify 而反向定义功能目标

## 5. Codex 执行约束（从下一轮开始）

### 5.1 批次声明

每个任务必须声明：

- 本轮是 `Batch 1/2/3`

### 5.2 Batch 3 封板

若 `Batch=3` 且 `PASS`：

- 必须标记 `CLOSED`
- 禁止继续提交 PR

### 5.3 继续优化门槛

若请求“继续优化”：

- 必须说明“新增用户价值”
- 无新增用户价值时，拒绝执行

## 6. 下一专题准入

仅在本规则执行完成后，允许进入：

- `ITER-1356` 用户价值优先能力
