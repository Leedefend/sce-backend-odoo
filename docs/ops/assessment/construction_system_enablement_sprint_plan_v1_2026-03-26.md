# 施工企业管理系统启用主路径冲刺计划 v1

- 日期：`2026-03-26`
- 状态：`Frozen for Execution`
- 适用阶段：`Phase 0 + Phase 1`

## 总体约束

- Layer Target：`Fact Layer + Contract Layer + Entry Layer + Interaction Layer + Verify Layer`
- Module：`addons/smart_enterprise_base + smart_construction_core 消费接线 + 必要的 Odoo 原生组织/用户模型扩展 + scripts/verify + docs`
- Reason：`把系统从可开发状态推进到可启用状态，并建立项目启动前置条件`
- 前端承接策略：`遵循 frontend_takeover_scope_v1；高频标准页前端承接，低频复杂页原生兜底`
- 不做：
  - 不做高级经营规则扩展
  - 不做新的风险/evidence 内核迭代
  - 不做跨阶段的审批平台化
  - 不做复杂多公司治理重构

## 页面分流硬规则

- 冻结依据：[frontend_takeover_scope_v1.md](/mnt/wsl/docker-desktop-bind-mounts/Ubuntu/0bf88c91312832ece483d20f9dd0da58b3449c7beac0658c5397b284fcec1f13/docs/product/frontend_takeover_scope_v1.md)
- 每个 Sprint 开工前，必须先标记本轮页面属于：
  - 前端承接
  - 原生保留
  - 条件承接
- 命中黑名单能力、契约不完整、依赖复杂原生行为的页面，统一走 `open_native`。
- 对原生保留页面，不再追加前端重绘任务，只允许补跳转、说明和兜底验证。
- 对前端承接页面，必须做到用户可见全链路闭环，不允许只补 contract 或只补 parser。

## 冲刺执行新原则 v1

- 每轮只做一个用户可感知主题。
- 每轮必须同时覆盖：
  - 事实层
  - 契约层
  - 入口层
  - 交互层
  - 验证层
- 不接受只落模型，不接受只落 verify。
- 没有用户路径演示，不算完成。
- 所有底层工作必须绑定到当前 Sprint 的用户动作上。

## 双验收标准

### 工程验收

- 模块 upgrade 通过
- verify 通过
- 不破坏旧链路

### 用户验收

- 必须能由人亲自走通：
  `进入 → 创建/修改 → 保存 → 查看结果 → 继续下一步`
- 只通过工程验收、不通过用户验收，视为未完成。

## 开工前四问

- 用户从哪里进入？
- 用户点什么完成操作？
- 操作成功后看见什么？
- 如果失败，系统怎么提示？

任一问题答不出来，不进入实现。

## 入口统一规则

- 每一个核心对象只允许一个主入口：
  - 公司
  - 部门
  - 用户
  - 项目
  - 任务
- 所有创建行为必须从该主入口发起。
- 允许存在快捷入口，但快捷入口只能跳转到主入口，不允许形成第二套创建语义。
- 若出现多个入口且行为不一致，本 Sprint 判定为未完成。
- 没有用户可见入口，不开工。
- 没有用户操作回路，不收口。

## Sprint 0

### 批次信息

- 批次：`Sprint-0`
- 目标：`建立业务主体与组织骨架，让系统具备最小可启用环境`
- 范围：
  - 公司基本信息
  - 组织架构
  - 最小表单与列表入口
  - 启用路径验证
  - 公司/部门前端承接与原生兜底边界固定
- 不做：
  - 用户/角色联动细节
  - 项目/任务逻辑

### 用户可执行动作（User Flow）

1. 使用 `admin / admin` 登录自定义前端
2. 进入“公司管理”主入口
3. 新建公司
4. 进入“组织架构”主入口
5. 新建部门树
6. 返回系统首页

用户验收标准：
- 用户无需阅读额外文档，能够完成以上动作
- 若命中复杂公司治理或复杂组织调整场景，系统应明确提供原生兜底路径

### 实施步骤

#### Step 1
- 操作：梳理公司主体最小字段与归属关系，确定后续项目/部门/用户统一挂载口径
- 修改范围：`smart_enterprise_base 公司模型扩展 + 表单/列表 + 选择默认公司语义`
- 输出：公司最小模型字段、视图、默认公司使用口径
- 页面归类：`前端承接（基础 list/form） + 原生保留（高级公司治理）`

完成判据：
- 能创建、编辑、启停公司
- 后续对象可以稳定关联公司

用户可见结果：
- 创建或编辑公司后，用户能立即看到保存结果
- 用户能明确识别“下一步去创建组织架构”

#### Step 2
- 操作：建立部门树最小闭环，支持公司归属、上级部门和负责人
- 修改范围：`smart_enterprise_base 中的 hr.department 扩展 + 树视图/表单 + 基础校验`
- 输出：至少支持三级组织结构
- 页面归类：`条件承接（基础维护前端，复杂组织调整原生）`

完成判据：
- 能建立部门树
- 能设置部门负责人
- 能按公司隔离部门

用户可见结果：
- 创建部门后，树结构立即可见
- 用户能从组织页判断当前处于哪一级组织

### 验证步骤
- verify：
  - 公司创建/编辑/启停 guard
  - 组织树创建 guard
- snapshot：
  - 公司/部门最小契约快照
- guard：
  - `公司 → 部门` 归属正确
- smoke：
  - 后台进入公司与组织入口无报错
  - 自定义前端首页出现“企业启用”卡片并可跳转到公司主入口

### 体验验收（UX Guard）

- 创建公司后有明确成功提示
- 创建部门后有明确成功提示
- 页面无空白页、无错误跳转、无无反馈保存
- 用户从首页能识别“下一步该去哪里”
- 保存失败时有明确错误提示，不出现白屏或静默失败
- `admin` 登录首页后，能看到唯一的企业启用主卡片，并从卡片进入公司管理
- 触发复杂治理场景时，不猜测渲染，系统明确给出原生兜底入口

### 风险与回滚
- 风险：错误改动 Odoo 原生公司/部门语义导致现有项目归属漂移
- 回滚：按模型扩展文件和视图文件回滚；必要时仅保留只读扩展字段

### 停机条件
- 无法明确“项目最终挂哪一个公司字段”
- 部门树无法与现有权限模型兼容

## Sprint 1

### 批次信息

- 批次：`Sprint-1`
- 目标：`建立可用的角色与用户基础，使系统真正有操作者`
- 范围：
  - 基础角色冻结
  - 用户设置最小闭环
  - 菜单/基础操作的角色差异
- 不做：
  - 复杂 RBAC 平台化重构
  - 精细到字段级权限治理

### 用户可执行动作（User Flow）

1. 进入“角色/权限”主入口
2. 查看基础角色集合
3. 进入“用户管理”主入口
4. 创建用户
5. 选择所属公司、所属部门、角色
6. 用新用户登录验证主入口差异

用户验收标准：
- 用户无需阅读额外文档，能够完成以上动作

### 实施步骤

#### Step 1
- 操作：冻结基础角色集合，明确每个角色最小入口和能力边界
- 修改范围：`group / capability / menu visibility / 最小角色矩阵`
- 输出：系统管理员、企业管理员、项目经理、部门负责人、普通员工、财务人员的最小角色定义

完成判据：
- 不同角色登录后主入口存在差异
- 至少项目经理与普通员工的入口明显不同

用户可见结果：
- 用户登录后能直接识别自己的角色差异
- 用户不会进入与自己无关的主入口

#### Step 2
- 操作：补齐用户设置最小字段，并建立公司/部门/角色归属
- 修改范围：`res.users` 扩展 + 关联字段 + 基础视图`
- 输出：可用测试用户集

完成判据：
- 能创建 3-5 个测试用户
- 项目/任务侧可选择这些用户

用户可见结果：
- 新建用户后，列表中立即可见
- 新用户登录后能看到匹配角色的首页和菜单

### 验证步骤
- verify：
  - 用户与角色关联 guard
  - 角色差异菜单 guard
- snapshot：
  - 用户最小资料契约
- guard：
  - 用户必须关联公司/部门/角色
- smoke：
  - 多角色登录入口差异手动或自动校验

### 体验验收（UX Guard）

- 用户创建成功后有明确提示
- 用户能清楚看到自己所属的公司、部门、角色
- 登录后菜单和主入口变化可被用户直接识别
- 不出现“账号已创建但无法判断自己拥有什么能力”的状态
- 缺公司、缺部门、缺角色时禁止保存，并给出明确提示

### 风险与回滚
- 风险：角色收敛过快导致现有功能被错误收口
- 回滚：优先回滚新角色矩阵和入口裁剪，不回滚已有基础 group

### 停机条件
- 无法冻结“角色真源”
- 用户字段与现有项目负责人/任务负责人选择链冲突

## Sprint 2

### 批次信息

- 批次：`Sprint-2`
- 目标：`建立项目创建与项目成员配置闭环`
- 范围：
  - 项目创建字段最小收口
  - 项目负责人/归口部门
  - 项目成员机制
- 不做：
  - 项目高阶治理字段
  - 项目经营分析

### 用户可执行动作（User Flow）

1. 进入“项目管理”主入口
2. 新建项目
3. 设置所属公司、归口部门、项目负责人
4. 保存后自动进入项目详情
5. 添加项目成员
6. 在详情页查看成员结果

用户验收标准：
- 用户无需阅读额外文档，能够完成以上动作

### 实施步骤

#### Step 1
- 操作：收口项目创建最小字段，明确公司、部门、负责人、起止时间
- 修改范围：`project.project` 扩展 + 创建/编辑视图 + 契约`
- 输出：项目最小创建闭环

完成判据：
- 新建项目后能进入详情
- 公司/部门/负责人清晰可见

用户可见结果：
- 创建项目后自动进入详情页
- 用户能在详情页确认项目归属和负责人

#### Step 2
- 操作：建立项目成员最小机制，并限制任务指派来源
- 修改范围：`project.project members + project/task selector constraints`
- 输出：项目成员列表与项目成员选择口径

完成判据：
- 一个项目能添加 2-3 个成员
- 任务负责人只能从项目成员中选择

用户可见结果：
- 成员添加后立即显示在项目详情
- 用户能明确识别下一步是“去创建任务”

### 验证步骤
- verify：
  - 项目创建 guard
  - 项目成员约束 guard
- snapshot：
  - 项目最小创建契约
- guard：
  - 项目成员必须来自用户主数据
- smoke：
  - 从项目列表进入项目详情无断链

### 体验验收（UX Guard）

- 创建项目后有成功提示
- 创建成功后自动进入项目详情，而不是停留在原页无反馈
- 添加成员后成员列表立即可见
- 用户能从详情页直接识别“下一步去创建任务”
- 缺公司、缺归口部门、缺负责人时禁止保存并给出提示

### 风险与回滚
- 风险：项目成员与现有 project.task 分派逻辑不兼容
- 回滚：保留成员显示层，暂时不收紧任务负责人约束

### 停机条件
- 项目归属公司/部门语义不稳定
- 项目成员与负责人口径冲突

## Sprint 3

### 批次信息

- 批次：`Sprint-3`
- 目标：`建立任务创建、分派与执行状态闭环`
- 范围：
  - 任务最小字段
  - 项目内创建任务
  - 分派负责人
  - 状态流转与执行备注
- 不做：
  - 高级任务协同
  - 复杂工时与绩效逻辑

### 用户可执行动作（User Flow）

1. 在项目详情进入“任务管理”主入口
2. 创建任务
3. 从项目成员中选择负责人
4. 保存任务
5. 被分派人登录后查看自己的任务
6. 把任务从未开始推进到进行中、已完成
7. 填写执行备注

用户验收标准：
- 用户无需阅读额外文档，能够完成以上动作

### 实施步骤

#### Step 1
- 操作：收口任务创建最小字段与项目内入口
- 修改范围：`project.task` 扩展 + 表单/列表 + 项目内创建入口`
- 输出：项目任务最小创建能力

完成判据：
- 能在项目下创建任务
- 列表可立即显示

用户可见结果：
- 新建任务后立即出现在项目任务列表
- 用户能确认任务已经挂在正确项目下

#### Step 2
- 操作：建立任务分派与状态流转闭环
- 修改范围：`负责人选择约束 + 状态字段/按钮 + 执行备注`
- 输出：任务从未开始到已完成的最小执行路径

完成判据：
- 被分派人能看到自己的任务
- 能从未开始推进到进行中、已完成

用户可见结果：
- 被分派人能在自己的任务入口直接看到新任务
- 状态变更后能明确看到最新状态与执行备注

### 验证步骤
- verify：
  - 任务创建 guard
  - 任务指派 guard
  - 任务状态流转 guard
- snapshot：
  - 任务最小执行契约
- guard：
  - 非项目成员不能被指派
- smoke：
  - 被分派人登录后可见自己的任务

### 体验验收（UX Guard）

- 创建任务后有成功提示
- 任务保存后立即出现在列表中
- 负责人能明确知道“这项任务已经分给我”
- 状态变更后有明确反馈，不出现用户误以为保存失败的情况
- 负责人不是项目成员时禁止保存，并给出明确提示

### 风险与回滚
- 风险：现有 task 扩展语义复杂，最小流转可能撞到既有状态机
- 回滚：在不破坏旧状态机前提下，保留兼容字段映射

### 停机条件
- 任务状态机语义无法最小收口
- 当前 `project.task` 已被其他流程强依赖且无法兼容

## Sprint 4

### 批次信息

- 批次：`Sprint-4`
- 目标：`把启用主路径收口为产品级可审核能力`
- 范围：
  - 启用路径引导
  - Demo/seed 校准
  - 验收文档
  - 总门禁
- 不做：
  - 新业务域扩展
  - 高阶驾驶舱功能

### 用户可执行动作（User Flow）

1. 新用户首次登录系统
2. 按引导依次完成公司、部门、用户、项目、任务配置
3. 进入项目详情
4. 创建并分派任务
5. 负责人登录并完成任务状态推进

用户验收标准：
- 用户无需阅读额外文档，能够完成以上动作

### 实施步骤

#### Step 1
- 操作：固化登录后启用路径，引导用户依次完成公司、部门、用户、项目、任务
- 修改范围：`入口编排 + 最小提示文案 + 启用路径 contract`
- 输出：统一启用主路径

完成判据：
- 新环境可沿统一路径完成初始化

用户可见结果：
- 新用户登录后能清楚看到当前处于启用流程的哪一步
- 每完成一步都能看到下一步入口

#### Step 2
- 操作：补齐验收包与总门禁
- 修改范围：`docs/ops/acceptance + scripts/verify + Makefile`
- 输出：`verify.product.enablement.v1`

完成判据：
- 启用路径总门禁通过
- 文档与系统行为一致

用户可见结果：
- 演示者可在不口头补充说明的情况下完成全流程
- 验收文档中的每一步都能在系统中直接找到入口和结果

### 验证步骤
- verify：
  - 总门禁 `verify.product.enablement.v1`
- snapshot：
  - 启用主路径 contract snapshot
- guard：
  - 种子环境从零到可运行全链校验
- smoke：
  - 新账号首次登录引导链路可走通

### 体验验收（UX Guard）

- 全流程无空白页、无死链、无无反馈保存
- 用户始终能识别当前处于哪一步、下一步该做什么
- 不需要口头解释即可完成启用主路径

### 风险与回滚
- 风险：引导路径与真实业务入口不一致，形成“假启用”
- 回滚：保留后台能力，先撤回引导入口与总门禁绑定

### 停机条件
- 启用路径无法与真实权限/数据状态保持一致
- 验收文档出现“文档可说、系统不可做”的漂移

- 只有在你确认后，才进入对应 Sprint 的实现批次。
- 默认执行顺序固定为：
  `Sprint 0 → Sprint 1 → Sprint 2 → Sprint 3 → Sprint 4`
