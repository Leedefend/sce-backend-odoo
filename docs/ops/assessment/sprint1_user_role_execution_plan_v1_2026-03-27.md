# Sprint 1 审核版执行计划 v1

- 日期：`2026-03-27`
- 状态：`Review Required`
- 目标阶段：`Phase 0 / Sprint 1`

## 1. 本轮唯一主题

把系统从“企业主体已就位”推进到“系统里真的有人，并且不同人登录后能看到不同主入口”。

本轮不是做权限平台，而是做：

```text
用户创建
→ 绑定公司/部门
→ 绑定基础角色口径
→ 登录后看到匹配入口
```

## 2. 架构定位

- Layer Target：`Fact Layer + Contract Layer + Entry Layer + Interaction Layer + Verify Layer`
- Module：`addons/smart_enterprise_base + frontend/apps/web + scripts/verify + docs`
- Reason：`在不重绘 Odoo 原生 ACL 的前提下，完成用户主数据与角色可见结果的最小闭环`

## 3. 页面分流判定

### 前端承接

- 用户列表
- 用户基础表单
- 登录后首页角色信息
- 登录后主入口差异提示

### 原生保留

- `res.groups` 复杂维护
- ACL / record rule / 高级菜单权限治理
- 多角色细粒度安全配置

### 条件承接

- 用户表单中的角色选择
说明：
- 只承接基础角色口径选择与展示
- 一旦命中复杂权限编辑，统一 `open_native`

## 4. 用户全链路定义

### 主路径 A：管理员创建用户

1. `admin / admin` 登录自定义前端
2. 从首页企业启用卡片或侧边导航进入“用户设置”
3. 点击“新建用户”
4. 输入姓名、登录账号、手机号
5. 选择所属公司、所属部门、直属上级
6. 选择基础角色口径
7. 点击保存
8. 返回用户列表，并看到新用户

### 主路径 B：新用户登录验证

1. 使用新建用户登录自定义前端
2. 看到自己的姓名、所属角色、所属公司
3. 看到与管理员不同的首页主入口
4. 能从首页进入本角色的第一个主工作入口

## 5. 体验验收（UX Guard）

- 用户列表必须有明确“新建用户”主按钮
- 用户保存成功后必须有明确提示
- 保存成功后必须回到用户列表或进入用户详情，不能停在无反馈状态
- 列表中必须立即出现新用户
- 登录后必须明确显示“当前角色/当前企业”
- 不同角色的首页入口必须有可见差异
- 复杂权限治理场景必须明确给出原生兜底入口
- 不允许白屏、空白详情页、无提示保存失败

## 6. 事实层范围

- `res.users` 最小字段集：
  - 姓名
  - 登录账号
  - 手机号
  - 所属公司
  - 所属部门
  - 直属上级
  - 基础角色口径
  - 状态
- 保持角色真源在 Odoo 原生 group，不新增前端权限真源

## 7. 契约层范围

- 用户列表 contract
- 用户基础表单 contract
- 登录后首页 `current_role/current_company/primary_entry` 最小展示 contract
- 用户保存成功反馈 contract
- 复杂权限场景 `open_native` 降级 contract

## 8. 入口层范围

- 首页到“用户设置”的唯一主入口
- 用户列表页“新建用户”唯一主按钮
- 用户保存成功后的返回路径
- 登录后首页角色信息展示入口

## 9. 交互层范围

- 必填校验：
  - 缺公司不能保存
  - 缺部门不能保存
  - 缺角色口径不能保存
- 成功反馈：
  - 保存成功 toast
  - 返回列表或进入详情
- 失败反馈：
  - 表单顶部错误提示
  - 不出现原始技术错误码

## 10. 验证计划

### 工程验收

- `smart_enterprise_base` 升级通过
- 用户基础 contract smoke 通过
- 不破坏 Sprint 0 的 `verify.product.enablement.sprint0`

### 用户验收

- 管理员能创建用户并在列表中立即看到
- 新用户能登录
- 至少两类角色用户登录后首页入口不同

### 建议门禁

- `make verify.product.enablement.sprint1`
- 包含：
  - 用户创建 guard
  - 用户列表/表单 smoke
  - 双角色登录首页差异 smoke

## 11. 本轮禁止事项

- 不做角色设计平台化
- 不做 group 编辑器
- 不做 ACL 可视化配置器
- 不做项目/任务逻辑
- 不做 evidence / risk / dashboard 扩展

## 12. 风险与停机条件

### 风险

- 用户字段与现有 Odoo 登录/权限链冲突
- 前端错误消费复杂权限结构，导致角色结果失真

### 停机条件

- 无法明确“角色真源仍在 Odoo 原生”这条边界
- 新用户登录后无法稳定拿到首页主入口差异
- 用户表单开始扩散到复杂安全配置

## 13. 本轮完成定义

只有同时满足以下条件，Sprint 1 才能进入实现完成态：

- 管理员能在自定义前端创建用户
- 用户必须绑定公司/部门/角色口径
- 新用户登录后能看到不同首页入口
- 复杂权限治理明确走原生兜底
- verify 与 smoke 同时通过
