# 业务管理员可视化配置中心 v1 · Phase 4 配置结果与契约联动预案

## 1. 目标

在不改变业务真相与既有冻结面的前提下，明确配置中心结果如何进入 contract 输出，以及哪些配置仅在后端参数层生效。

## 2. 联动总原则

- 先后端供给，再前端消费。
- 仅允许 non-breaking、可选扩展输出进入 contract。
- 任何会改变权限/审批/业务事实判断的配置，不进入 contract 配置输出面。

## 3. 配置结果分层

## 3.1 Contract 可选输出面（候选）

面向前端消费的配置结果：
- `navigation.entries`（角色入口显隐与排序建议）
- `home.blocks`（首页区块显隐、分组、排序建议）
- `display.hints`（文案或轻量渲染提示）

约束：
- 仅作为可选扩展字段。
- 不覆盖既有冻结字段语义。
- 缺省时前端回退到既有稳定行为。

## 3.2 Backend-only 参数面

只在后端生效，不进入 contract 主体输出：
- 默认分页大小
- 默认排序偏好
- 默认布局策略
- 管理端行为开关（非业务规则）

约束：
- 不影响权限判定与业务流程语义。
- 不作为前端业务决策唯一依据。

## 4. 冻结策略（v1）

冻结对象：
- `contract freeze surface` 既有字段保持不变。
- 新增字段先进入 `candidate` 状态，不立即冻结。

冻结流程：
1. 候选字段登记（字段名/语义/供给条件）。
2. 运行时样本验证（按角色抓取 payload）。
3. 前端真实消费验证（确认无兜底掩盖）。
4. 独立验收批次通过后进入冻结面。

## 5. 非破坏发布策略

发布策略：
- 阶段 1：后端供给但默认关闭（feature flag）。
- 阶段 2：灰度角色启用并抓取运行时证据。
- 阶段 3：全量开启并更新 freeze 文档。

回滚策略：
- 配置开关回滚到关闭态。
- 前端保持旧行为，不依赖新增字段。

## 6. 前端消费约束

- 前端不得假设配置字段必然存在。
- 前端不得使用配置结果替代权限判定。
- 前端不得新增模型特判去“修复”配置缺失。

## 7. 联动验收清单（后续执行）

最低验收项：
- Contract payload 提供候选字段时，shape 稳定。
- 字段缺失时，前端路径仍可工作。
- outsider deny 语义不受配置影响。
- 原生最小办理链不受配置联动破坏。

## 8. 结论

- `PASS`（预案阶段）：配置结果与契约联动边界已定义，具备进入后续实现批次的治理基础。
- 本专题 v1 启动目标达成：Phase 1~4 的规划基线完整且互相衔接。
