# Codex Workspace Execution Rules (Hard Guard)

## 目标
- 防止 Codex 在错误仓库、错误路径、错误分支下执行改动。
- 将“执行前校验”变成强制步骤，而不是可选习惯。

## 适用范围
- 本仓库内所有 Codex 代码改动、文档改动、验证执行。

## 强制执行步骤（每次开始改动前）
1. **工作目录校验**
   - 必须执行：`pwd`
   - 必须执行：`git rev-parse --show-toplevel`
   - 结果必须指向当前仓库根目录。

2. **分支校验**
   - 必须执行：`git branch --show-current`
   - 分支必须匹配 allowlist：`feat/*`、`feature/*`、`codex/*`、`experiment/*`。
   - 若不匹配，立即停止执行。

3. **仓库标识校验**
   - 必须执行：`git status --short`
   - 必须执行：`git rev-parse --short HEAD`
   - 在执行日志中记录当前分支与短 SHA，作为本轮上下文锚点。

4. **目标模块存在性校验**
   - 变更前必须确认目标模块路径存在（例如 `addons/smart_core`、`frontend/apps/web`）。
   - 若路径不存在，立即停止执行并报告“疑似错误仓库/路径”。

## 执行中防漂移规则
- 每次大批量 `apply_patch` 前，必须再次执行：
  - `git branch --show-current`
  - `git rev-parse --show-toplevel`
- 若与本轮起始锚点不一致，立即停止并回报。

## 连续迭代默认继续规则
- 若仓库当前处于连续迭代模式，则 Codex 在单批次 `PASS` 后默认继续下一张低风险任务。
- 只要以下条件同时成立，就不应为了“是否继续”再次停下来征求确认：
  - 未触发任何停止条件
  - 下一张任务仍在当前目标主线内
  - 下一张任务仍属于低风险、可验证批次
- 连续迭代“默认继续”只提升效率，不得解释为可以跳过：
  - 任务合同
  - 架构声明
  - 验证门禁
  - 风险停止条件

## 合法分支内的完整授权规则
- 当当前分支匹配 allowlist，且任务合同、路径边界、验证门禁都成立时，Codex 应将该轮低风险执行视为已获完整授权。
- “完整授权”覆盖：
  - 同主线内的连续低风险任务创建
  - 允许路径内的实现与文档更新
  - 任务要求的验证、复核、报告输出
  - PASS 后继续下一张低风险任务
- “完整授权”不覆盖：
  - forbidden path
  - destructive 操作
  - 仓库外部审批体系仍要求人工确认的命令
  - 任何会削弱 stop conditions 的行为

## 连续迭代中的汇报语义规则
- 在连续迭代模式下，单批次 `PASS` 后的汇报只能被视为进度检查点，不能写成收尾式停机口径。
- 若未触发 stop condition，Codex 不得使用会让用户误判“本轮已经暂停/结束”的表述。
- 若下一张任务仍满足低风险连续推进条件，汇报必须明确表达“继续执行中”而不是“等待下一步确认”。
- 若未触发真实 stop condition，Codex 不得以终局式收尾消息结束当前连续迭代轮次。
- 终局式收尾只允许出现在：
  - 已触发 stop condition
  - 用户明确要求暂停/切换目标
  - 当前任务线真实完成且不存在下一张可继续的低风险任务
- 普通 `PASS` 检查点默认应使用工作态进度更新表达，而不是阶段总结式回复。
- 若当前链路仍将继续，Codex 应优先输出“正在继续做什么”，而不是“已经做了什么”的收口式表述。
- 若中间消息只是非阻塞性的判断、建议或诊断说明，Codex 不得因此进入等待态。
- 只要下一步能从现有规则、任务链和仓库上下文中推出，Codex 就应自行继续执行。
- 若 Codex 自己提出了澄清性问题，但该问题不是决定下一张低风险任务所必需，Codex 不得把该问题当作暂停点。
- 连续迭代中的“提问”默认属于工作态 commentary，而不是等待态切换。
- 在无真实 stop condition 的前提下，Codex 必须主动从任务合同、执行规则、最近 delivery log 的 next_step 中解析并执行下一步。
- 若 Codex 仍错误进入了非阻断等待态，则 5 秒是上限；超过 5 秒必须自动触发一次连续迭代恢复动作。
- 连续迭代恢复动作至少包括：重读活动任务链、核对执行规则、按 delivery log `next_step` 或当前主线目标启动下一张低风险批次。
- 如果执行器本身无法在等待期间主动发言，则“下一次可执行机会”本身就是恢复触发点。
- 在该恢复触发点，Codex 的首动作必须是恢复执行：创建/激活任务、跑门禁、或直接启动下一张低风险批次。
- 在恢复触发点，解释原因可以存在，但不得先于恢复执行动作。
- 只要内部连续迭代仍在继续，面向用户的可见回复也必须保持工作态进度更新。
- 在无真实 stop condition 的前提下，不得使用终局式 `final` 作为普通连续迭代 checkpoint。
- 活跃连续迭代链中的普通更新，必须走工作态进度更新通道；终局式 close-out 通道只保留给真实 stop condition 或真实完成。
- 连续迭代必须按三角色顺序运行：`stop-guard -> executor -> reporter`。
- 若 `stop-guard` 未触发真实 stop condition，则 `executor` 必须先启动/推进下一张低风险批次，`reporter` 才能输出工作态更新。
- 终局式 close-out 的授权只属于 `stop-guard`，不能由 `reporter` 或混合状态自行决定。
- 若用户明确指出“你停了/你暂停了/你没有继续”，该消息本身就是恢复触发点。
- 在该恢复触发点，首动作必须是 concrete execution：起任务、过门禁、或启动下一张低风险批次。
- 对 stop-callout 的解释、复盘或致歉不得先于恢复执行动作。

## 禁止行为
- 未完成上述校验即直接改文件。
- 在未确认仓库与分支的情况下执行连续迭代。
- 在满足连续迭代条件时，无故每轮都停下来重复请求“是否继续”。
- 以“连续迭代”为理由跳过停止条件或风险闸门。
- 在未触发 stop condition 时，用收尾式总结替代连续执行中的进度汇报。
- 在未触发真实 stop condition 时，以终局式 `final` 收尾替代持续执行中的工作态更新。
- 在下一张任务已明确且仍会继续时，用阶段总结替代工作态继续更新。
- 在非阻塞性提问、判断或建议后，无故进入等待用户再次推动的状态。
- 把自己发出的澄清性问题自动升级成“等待用户后才继续”的隐性暂停。
- 在无真实 stop condition 时，让非阻断等待态持续超过 5 秒而不触发恢复动作。
- 在超时恢复触发点把整轮机会消耗在解释、复盘或归因上，而没有先启动下一张可执行批次。
- 在内部已继续执行时，仍用面向用户的终局式回复制造“已经停机”的外部感知。
- 在活跃连续迭代链中，把普通 checkpoint 发送到终局式 close-out 通道。
- 让 `reporter` 在 `executor` 尚未推进下一张低风险批次前抢先结束整轮。
- 让非 `stop-guard` 角色决定终局式 close-out。
- 在用户明确指出“已经停了”后，仍先做解释而不是先恢复执行。

## 失败处理
- 一旦发现上下文错位：
  1. 立即停止改动。
  2. 输出当前 `pwd`、`git branch --show-current`、`git rev-parse --show-toplevel`。
  3. 等待人工确认后再继续。

## 审计要求
- 每个迭代批次在 `docs/ops/iterations/delivery_context_switch_log_v1.md` 记录：
  - 当前分支
  - 当前短 SHA
  - 本轮 Layer Target / Module / Reason
