# 项目调度总体原则 v1（Project Dispatch Principles v1）

## 1. 文档目标

本文件用于统一项目调度、技术决策与功能迭代方向，确保系统在以下三大目标约束下持续演进：

1. **快速交付第一个可用产品（First Deliverable）**
2. **最大限度复用 Odoo 原生能力（Native-first）**
3. **保持既有架构边界稳定（Architecture Integrity）**

本原则适用于：

- 功能迭代决策
- 架构设计判断
- 任务优先级排序
- Codex / 自动化调度输入基准

---

## 2. 当前系统阶段定位

当前系统处于：

```text
产品生命周期主链已建立（initiation → dashboard → plan → execution）
+ 最小执行推进能力已建立（execution.advance）
+ 启动分层与 runtime fetch 架构稳定
```

系统性质已从：

> 功能系统 → 流程系统 → 可推进系统

---

## 3. 核心调度原则（必须同时满足）

### 原则一：以“首个可交付产品”为最高目标

所有调度必须优先回答：

> 本次迭代是否让“第一个可交付产品”更接近完成？

不满足该条件的任务，默认降级优先级。

### 原则二：优先解决“阻塞交付”的问题

只优先推进两类任务：

1. 阻塞产品链路走通的问题
2. 影响产品稳定性/可演示性的问题

其他优化（性能、优雅性、扩展性）如不影响交付，延后处理。

### 原则三：严格控制横向扩张

当前阶段禁止优先推进：

- 新大场景
- 新模块体系
- 新抽象层设计
- 平台级重构

除非该任务直接服务首个产品交付。

### 原则四：每轮调度必须具备“交付增量”

每一轮任务必须带来以下至少一项提升：

- 用户可见能力增强
- 产品链路更完整
- 体验更可理解
- 演示路径更顺畅

### 原则五：优先“产品化封装 Odoo”，而不是“重写 Odoo”

系统定位：

> 基于 Odoo 的产品化引擎，而不是替代 Odoo 的系统

必须复用的 Odoo 原生能力：

- ORM / 模型体系
- ACL / Record Rule 权限体系
- chatter / mail.thread（协同）
- activity（待办）
- attachment（附件）
- server action / automated action
- 项目 / 任务（project.task）
- 审计与消息机制

禁止行为：

- 重建权限系统
- 重建消息/协同系统
- 重建审批体系（如已有可用）
- 重建附件/文件体系
- 重建任务/状态体系（如 Odoo 已可承载）

### 原则六：产品层负责“组织流程”，Odoo 负责“承载数据”

职责划分：

| 层级 | 职责 |
| --- | --- |
| Odoo 原生层 | 数据、权限、协同、审计 |
| 产品层（当前系统） | 流程组织、用户引导、场景编排 |

### 原则七：架构边界不可回退

必须持续遵守：

```text
entry（最小）
+ runtime blocks（按需加载）
+ suggested_action（流程驱动）
+ contract（结构约束）
```

禁止行为：

- 将 runtime 数据回灌到 `system.init`
- 将 entry 变成大聚合接口
- 新建绕过 contract 的数据通路

### 原则八：所有能力必须可验证（Verification First）

新增功能必须具备：

- contract guard
- flow smoke
- baseline 集成

禁止：

> “功能存在但不可验证”

## 4. 当前产品定义（v0.1）

### 产品名称（建议）

> 智能工程项目全生命周期工作流（v0.1）

### 当前能力范围

```text
立项（initiation）
→ 驾驶舱（dashboard）
→ 计划准备（plan_bootstrap）
→ 执行（execution）
→ 执行推进（execution.advance）
```

### 产品特点

- 可进入
- 可流转
- 可推进
- 可验证

---

## 5. 当前阶段优先事项

### 第一优先级：交付收口（Delivery Closure）

- 页面可理解（文案/状态/提示）
- 流程清晰（每一步都有“下一步”）
- 成功/阻塞反馈清晰

### 第二优先级：产品基线冻结

- 固定 v0.1 功能范围
- 固定 flow baseline
- 固定 contract 结构
- 输出验收路径

### 第三优先级：Execution 最小体验闭环

- 推进后有反馈
- 状态变化可感知
- 用户知道“接下来做什么”

---

## 6. 调度决策三层判断模型（必须使用）

每个任务必须通过以下三层判断：

### 第一层：是否加速交付？

→ 否：降级

### 第二层：是否复用 Odoo 原生能力？

→ 否：优先重构方案

### 第三层：是否破坏当前架构边界？

→ 是：禁止执行

---

## 7. 当前禁止优先推进的事项

- 深度状态机（复杂规则）
- AI 编排
- 多角色复杂权限矩阵
- 大规模 block 扩展
- 新产品链启动
- 平台层大重构

---

## 8. 当前允许推进的事项

- execution 轻量增强（不破坏结构）
- 用户体验优化
- flow 连续性优化
- 文档与交付说明
- Odoo 原生能力对齐

---

## 10. 结论

当前阶段的核心不是：

> “继续构建更复杂系统”

而是：

> **把已有系统收口为第一个可交付产品，并确保它站在 Odoo 原生能力之上。**

---

## 11. 调度口令（用于快速校准）

每次决策前，问三句话：

1. 这是否让第一个产品更接近交付？
2. 这是否在复用 Odoo，而不是绕开它？
3. 这是否保持了当前架构边界？

如果有任一答案为“否”，则该任务不应优先执行。
