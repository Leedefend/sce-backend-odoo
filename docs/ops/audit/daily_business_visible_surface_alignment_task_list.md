# 用户日常业务可见面对齐任务清单

## 来源与边界

- 来源文档：`/home/odoo/workspace/partner_import_source/老系统列表，填单页面截图.docx`
- 提取方式：docx 内嵌截图人工转写，截图资源临时解包于 `/tmp/old_system_docx/word/media/image*.png`。
- 本清单目标：把用户日常办理入口的列表字段、默认顺序、表单字段、子表、附件和日志统一整理成后续系统对齐任务。
- 架构边界：字段顺序、表单分区、字段显隐、默认排序、筛选项、附件区、日志区必须由 Odoo 原生视图和后端契约输出；自定义前端只消费契约并通用渲染，不在前端硬编码业务字段。

## 统一验收口径

每个业务入口完成时必须同时满足：

1. 列表契约包含老系统截图中的字段顺序、默认筛选、状态分组和默认排序。
2. 表单契约包含业务分区、x2many 子表、附件区、日志/chatter 区、状态栏和按钮行为。
3. 字段缺失先回到模型或原生 XML 补齐，再由契约解析层输出；不得让前端根据页面名称推断字段。
4. 所有日常业务单据默认具备附件和日志能力；旧系统截图未显示日志的，也按新系统统一结构补齐。
5. 验收必须覆盖运行态契约审计和浏览器真实路径验证。

## 实施分层

| 层级 | 职责 | 本批任务要求 |
| --- | --- | --- |
| 业务事实/模型层 | 承载字段、状态、关系、附件、日志 | 缺字段时先补模型字段或专业承载模型，不用前端补假字段 |
| 原生视图层 | 定义列表列序、表单 group/notebook、x2many、search | 对齐老系统字段顺序和分区，作为契约来源 |
| 契约解析层 | 输出标准 list/form/search/action contract | 补齐状态、权限、字段显隐、默认排序、附件和日志语义 |
| 自定义前端 | 通用渲染契约 | 不写业务页面特例，不手工维护字段顺序 |
