# 数据迁移重建目标 v1

任务：`ITER-2026-04-13-1847`

## 目标重定向

数据迁移总体目标调整为：

> 在完整新库中可重复、可审计、可回滚地重构旧系统业务数据。

这意味着后续迁移不是一次性把 CSV 或旧库数据“塞入当前库”，而是形成一套可重复执行的重建流水线。

## 新目标

最终迁移链路必须满足：

1. 可从空的新库开始；
2. 可按固定顺序重建主数据和业务单据；
3. 每一步都有稳定的 legacy identity；
4. 每一步都能 dry-run；
5. 每一步都能给出 create/update/skip/reject 结果；
6. 每一步都有精确回滚锁定键；
7. 每一步都能重复运行而不产生重复数据；
8. 每一步都能输出审计报告。

## 当前脚本定位

当前已生成的迁移脚本统一视为：

- 源数据探针；
- 字段覆盖审计；
- 映射 dry-run；
- 候选表生成；
- 写入前验证工具。

它们不是最终正式导入脚本，除非后续通过脚本晋级门禁。

## 导入顺序原则

后续重建顺序应优先遵循：

1. 系统基础主数据；
2. partner 主数据；
3. 项目骨架；
4. 合同骨架；
5. 合同与项目/partner 的关系回填；
6. 后续付款、回款、结算、附件等业务链路。

当前合同链路必须继续被 partner readiness 阻断，不能绕过 partner 主数据去导合同。
