## 目标

把 `docs/migration_alignment/scbs_55_user_visible_surface_live_alignment_v1.md`
中的 55 个用户可见入口，从“字段/入口对齐”推进到“业务数据可交付”。

本计划的交付标准不是表存在、字段存在、页面能打开，而是用户以旧系统口径打开对应入口时：

- 列表总数、筛选后数量、关键样本单据可解释。
- 用户可见列来自旧系统对应主表/从表/报表 SQL，不用目标表猜值。
- 金额、税额、日期、状态、附件、录入人和录入时间有源头证据。
- 附件显示为用户可识别的文件名/链接，不直接裸露 hash。
- 新系统 action/domain 只显示该入口应显示的数据，不混入同模型其他业务口径。

## 总原则

1. 不直接在目标业务表随手补数。先定位旧库源表、迁移资产、事实表、投影表、用户可见 action 五层中的断点。
2. 主表/从表字段来源以 55 清单中的 `main_table`、`from_table`、`visible_columns` 为准。
3. 报表入口必须补旧系统报表 schema/SQL 口径，不能只按现有新系统汇总模型凑数。
4. 大屏入口必须定义指标字典、过滤条件、维度和钻取来源；只有行数不等于大屏合格。
5. 每修一个入口，必须形成一条可复跑验收证据：旧库计数、源事实计数、目标计数、样本比对、浏览器截图或页面证据。

## 五层验收门

每个入口必须依次通过下面五层，任何一层失败都不能交付给用户。

| 层级 | 验收内容 | 合格标准 |
| --- | --- | --- |
| A. 旧库口径 | 旧系统菜单、ConfigId、主表、从表、报表 SQL、默认排序、默认过滤 | 55 清单字段齐全；报表/自定义页补 schema；能在旧库只读 SQL 复现列表总数 |
| B. 迁移资产 | 资产包是否包含对应源表和字段 | 源表行数、关键字段非空率、附件索引数量可复跑 |
| C. 事实层 | `sc_legacy_*` 或专用 fact 是否按旧口径承载 | 事实行数与旧库口径一致或差异有明确过滤原因 |
| D. 投影层 | 新系统正式模型是否只投影该业务入口应有数据 | 目标 action/domain 数量正确；金额/日期/状态/附件非空率符合旧库 |
| E. 用户可见层 | 列表、搜索、表单、附件、样本单据 | 用户可见列、顺序、名称、值、总数、样本均通过 |

## 旧库上传完成后的执行步骤

1. 冻结旧库只读连接参数和数据库名，先不跑任何写入脚本。
2. 跑 55 清单旧库复核：菜单入口、ConfigId、源表、旧库总数、可见列。
3. 跑迁移资产覆盖检查：源表是否进入资产包，字段是否完整。
4. 跑当前新库只读质量探针，生成差异清单。
5. 按 P0 -> P1 -> P2 -> P3 -> P4 逐项修适配器/事实层/投影/action。
6. 每修完一个入口，跑单入口验收；通过后标记 `PASS`，未通过保持阻断。
7. 55 个全部 PASS 后，才进入用户直接使用验收。
