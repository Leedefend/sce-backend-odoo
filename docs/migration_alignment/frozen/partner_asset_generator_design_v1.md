# Partner Asset Generator Design v1

Status: Frozen Migration Governance
Depends on:

- `docs/migration_alignment/frozen/legacy_business_fact_asset_rebuild_plan_v1.md`
- `docs/migration_alignment/frozen/migration_asset_manifest_schema_v1.md`
- `docs/migration_alignment/partner_mapping_strategy_v1.md`

Objective: define the no-DB partner asset generator contract. This is a design batch only. It does not write to Odoo, does not create `res.partner`, and does not modify existing migration scripts.

## 1. Position in the new migration mainline

The partner lane is the first sample lane for the assetized rebuild bus:

```text
legacy partner facts
  -> normalized partner facts
  -> target-model partner asset files
  -> manifest + external id manifest + validation manifest
  -> later fresh DB loader
```

The generator output must be reusable and deterministic. A rerun with the same source snapshot and rules must produce the same records and stable external ids.

## 2. Generator boundary

Script to implement in the next batch:

```text
scripts/migration/partner_asset_generator.py
```

Allowed behavior:

- Read raw partner source files.
- Normalize partner facts according to frozen mapping.
- Generate asset package files under a runtime or generated asset directory.
- Generate manifest files.
- Generate discard summary.
- Run no-DB validation.

Forbidden behavior:

- No Odoo shell.
- No database write.
- No `res.partner` create/update/unlink.
- No merge side effects.
- No payment, settlement, accounting, ACL, record rule, or manifest changes.
- No mutation of existing source CSV files.

## 5. Normalized partner facts

The generator must first produce an in-memory normalized shape:

| Field | Source | Required | Rule |
| --- | --- | --- | --- |
| `legacy_partner_key` | company `Id`, supplier `ID`, or canonical key | yes | stable key for external id |
| `source_table` | `company` or `supplier` | yes | source provenance |
| `source_role` | derived | yes | `company`, `supplier`, or `combined` |
| `partner_name` | company `DWMC`, supplier `f_SupplierName` | yes | trim whitespace |
| `tax_number` | company `SH` then `TYSHXYDM`; supplier `SH`, `SHXYDM`, `NSRSBH`, `TISHXYDM` | no | normalize empty to blank |
| `phone` | company `YWLXRHM`, supplier `f_Phone` | no | normalize empty to blank |
| `email` | company `YWLXRYX`, supplier `f_Email` | no | normalize empty to blank |
| `customer_flag` | company source | yes | `true` for company facts |
| `supplier_flag` | supplier source evidence | yes | `true` for supplier facts |
| `discard_reason` | derived | conditional | required for discarded facts |

Garbage records:

- Missing `partner_name`: discard.
- Missing stable `legacy_partner_key`: discard.
- Unsafe empty enterprise identity after normalization: discard.
- Non-company personal fragments: discard unless already mapped to an enterprise canonical key.

## 6. Target asset fields

`partner_master_v1.csv` columns:

```text
external_id,legacy_identity_key,legacy_partner_key,source_role,name,is_company,vat,phone,email,customer_rank,supplier_rank
```

Rules:

- `external_id`: generated by the stable external id rule.
- `legacy_identity_key`: `partner:<source>:<legacy_partner_key>`.
- `name`: normalized `partner_name`.
- `is_company`: always `true`.
- `vat`: normalized `tax_number`, optional.
- `phone`: optional.
- `email`: optional.
- `customer_rank`: `1` when customer evidence exists, otherwise `0`.
- `supplier_rank`: `1` when supplier evidence exists, otherwise `0`.

No other target field is required in the first asset generator. New system non-required fields may remain empty.

## 7. Stable external id

Pattern:

```text
legacy_partner_sc_<legacy_partner_key>
```

Example:

```text
legacy_partner_sc_3308
```

Rules:

- The prefix is fixed for this lane: `legacy_partner_sc`.
- The suffix must be the stable legacy key or frozen canonical key.
- The value must not contain a new database id.
- Duplicate external ids block the package.
- Discarded records do not enter `partner_master_v1.csv`, but their counts and reasons must enter `partner_discard_summary_v1.csv`.

## 8. Dedup and conflict policy

The generator design does not authorize automatic writes or merges. It only classifies.

Classification:

| Class | Asset status | Rule |
| --- | --- | --- |
| `loadable` | included in `partner_master_v1.csv` | stable key and required name exist |
| `discarded` | excluded, counted in discard summary | garbage or required identity missing |
| `conflict_blocked` | excluded or deferred | same tax/name conflict not safely canonicalized |
| `deferred` | excluded from first package | needs later dependency or manual rule |

Conflict handling:

- Same external id with different normalized name: block package.
- Same tax number with conflicting enterprise names: `conflict_blocked`.
- Same normalized name with no tax number: allowed only when stable legacy key differs; no merge in generator.
- Supplier/company combined facts may share one canonical key only if the canonical key was frozen before generation.

## 9. Manifest outputs

### partner_asset_manifest_v1.json

Must conform to `asset_manifest_version = 1.0`.

Required package values:

```json
{
  "asset_manifest_version": "1.0",
  "asset_package_id": "partner_sc_v1",
  "lane": {
    "lane_id": "partner",
    "layer": "10_master",
    "business_priority": "core_master",
    "risk_class": "normal"
  },
  "target": {
    "model": "res.partner",
    "identity_field": "legacy_identity_key",
    "load_strategy": "upsert_by_external_id"
  }
}
```

### partner_external_id_manifest_v1.json

Required record shape:

```json
{
  "external_id": "legacy_partner_sc_3308",
  "legacy_key": "3308",
  "legacy_key_type": "single_pk",
  "source_table": "company",
  "target_model": "res.partner",
  "target_lookup": {
    "field": "legacy_identity_key",
    "value": "partner:sc:3308"
  },
  "status": "loadable"
}
```

### partner_validation_manifest_v1.json

Must list validation_gates for:

- `source_snapshot_declared`
- `external_id_unique`
- `required_fields_present`
- `asset_files_exist`
- `asset_hashes_match`
- `no_duplicate_business_identity`
- `no_high_risk_lane_leakage`

## 10. no-DB validation_gates

The implementation batch must validate all of these without an Odoo database:

1. Source files exist and are readable.
2. Required source headers exist.
3. Output directories are created under the requested `--out`.
4. `partner_master_v1.csv` has the required header.
5. `external_id` values are unique.
6. `legacy_identity_key` values are unique among loadable rows.
7. Required target fields are present for loadable rows.
8. Discarded rows have a discard reason.
9. Manifest `record_count` matches generated CSV rows.
10. Manifest hashes match generated files.
11. No asset is emitted for payment, settlement, accounting, ACL, record rules, or module manifests.

The next implementation must stay incremental:

## 13. next implementation stop conditions

Stop if:

- Source headers differ from the frozen mapping.
- External id uniqueness cannot be guaranteed.
- A required target field is missing for a loadable row.
- The generator needs Odoo ORM to decide facts.
- The implementation attempts to create or update `res.partner`.
- The batch starts touching contract, project, receipt, payment, settlement, accounting, ACL, record rules, or module manifests.

## 14. next implementation output policy

Generated files under `.runtime_artifacts/` remain local runtime evidence.

The next implementation should commit only:

- `scripts/migration/partner_asset_generator.py`
- Any formal schema or runbook update that is necessary.

It should not commit generated `partner_master_v1.csv` or manifest JSON unless a later stage promotes a frozen baseline summary.
