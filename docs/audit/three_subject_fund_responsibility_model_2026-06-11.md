# 三主体资金责任模型 - 2026-06-11

## 结论

用户旧系统没有统一建立“往来款”概念，不能按旧菜单或旧表名直接切分新系统口径。资金域必须先按三个主体识别业务事实：

- 公司
- 项目
- 承包人

项目不总是往来对方。很多场景里，项目是资金状态和业务约束维度；真正的责任关系发生在公司与承包人之间。

## 三类事实

| 事实层 | 典型来源 | 主体关系 | 项目作用 | 办理用途 |
| --- | --- | --- | --- | --- |
| 项目往来事实 | 项目借公司款、项目还公司款、承包人借项目款、承包人还项目款、项目间/账户间调拨 | 公司-项目、项目-承包人、项目-项目 | 往来主体之一或调拨锚点 | 借款、还款、调拨、现金流台账 |
| 公司-承包人责任事实 | 到款确认、自筹垫付、自筹退回 | 公司-承包人 | 项目资金状态、归集和约束维度 | 判断可拨付、应扣、应退、垫付占用、后续付款/借还约束 |
| 资金状态输入 | 资金日报、账户收支明细、余额调整、项目资金余额 | 不直接产生新责任 | 项目/账户状态 | 约束办理动作、辅助核对台账和余额 |

## 到款确认

到款确认不是简单的“收入事实”，也不能排除在往来责任之外。它的用户业务含义是：

- 反映某个项目的回款状态。
- 记录公司收到或确认的项目资金。
- 记录代扣代缴、拨付金额和留存。
- 通过 `construction_unit_name` 指向承包人/责任人。
- 后续约束公司对承包人的拨付、扣款、退回、自筹抵扣和项目资金可用状态。

本地 `sc_demo` 数据：

| 指标 | 数值 |
| --- | ---: |
| 到款确认记录 | 5205 |
| 关联项目 | 5200 |
| 有承包人/责任人名称 | 5085 |
| 到款金额 | 4,504,749,129.91 |
| 代扣代缴 | 207,631,365.44 |
| 拨付金额 | 4,323,349,159.76 |

因此，到款确认应进入“公司-承包人责任事实”，同时继续按项目归集，作为项目资金状态和后续办理约束来源。

## 自筹垫付

自筹类反映的是承包人与公司的资金占用和归还关系，不是普通项目收入。项目用于说明这笔自筹归属哪个项目、如何影响项目资金状态和后续办理约束。

本地 `sc_demo` 正式余额族：

| 来源族 | 数量 | 金额 |
| --- | ---: | ---: |
| 自筹收入 `income` | 2153 | 223,309,983.61 |
| 自筹退回 `refund` | 1575 | 145,077,200.09 |
| 自筹未退净额 | - | 78,232,783.52 |

可见验收族：

| 来源族 | 数量 | 说明 |
| --- | ---: | --- |
| 自筹垫付收入 `income_visible` | 2144 | 用户旧入口可见记录 |
| 自筹垫付退回 `refund_visible` | 827 | 用户旧入口可见记录，金额不作为余额计算 |

因此，自筹应进入“公司-承包人责任事实”。正式余额使用 `income/refund`，旧入口验收使用 `income_visible/refund_visible` 作为认知和追溯参考。

## 当前实现偏差

上一版边界审计把到款确认、自筹放在“不直接纳入往来款”下，这会误导后续迭代。正确表达应是：

- 不纳入 `sc.interfund.movement.fact` 的项目借还调拨事实层。
- 但必须纳入公司-承包人责任事实层。
- 继续保留在 `sc.finance.business.fact` 和项目资金汇总里，但分类语义要从“收入/自筹事实”提升为“公司-承包人责任事实 + 项目资金约束”。

## 后续实现方向

1. 保留用户旧入口认知：到款确认表、自筹垫付收入、自筹垫付退回。
2. 在业务分类字典中增加或修正责任分类：公司-承包人到款责任、自筹垫付、自筹退回。
3. 建立只读责任投影，按公司、承包人、项目汇总到款、扣款、拨付、自筹、退回、未退和可用余额。
4. 用项目资金状态约束后续办理：付款、借款、还款、退回、自筹抵扣、收款发票核销。
5. 不把这些事实强行挂到 `payment.request`，但发生真实现金流时仍应进入统一资金台账或日报台账口径。
