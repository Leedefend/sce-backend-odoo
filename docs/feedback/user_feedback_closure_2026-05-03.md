# 5.3 用户反馈闭环计划

来源文件：`/home/odoo/workspace/5.3日优化.docx`

本文档沉淀本轮从用户反馈 Word 截图中提炼出的业务问题、已完成改动和后续迭代步骤，后续推进以本文档为基准，避免重复拆解原始附件。

## 处理原则

- 业务事实层只承载模型、视图、权限、菜单，不把流程编排、模拟数据、演示逻辑放回业务事实层。
- 新系统模型要承接历史数据字段，优先用正式模型字段表达历史事实；仅无法归入正式模型的内容保留为历史事实模型。
- 审批不硬阻断业务风险。业务规则不满足时应提示给审批人，由审批人判断是否同意推进；系统只对数据闭环、状态一致性、权限边界做硬约束。
- 所有字段补强都要同时覆盖模型、表单视图、列表视图、访问权限和升级/重建可复现性。
- 对“级联创建、必填但界面不可见、技术默认缺失”导致的保存失败，采用模型层系统默认兜底策略：不取消业务字段语义上的必填，不阻断草稿创建；由系统写入“系统默认（待完善）”值并打标识，真实业务推进前由业务人员补全。

## 系统默认兜底策略

适用范围：

- 级联创建时由父单或来源单应自动带出的字段。
- 技术上 required，但当前表单、弹窗或导入链路不可见/不可编辑的字段。
- 草稿阶段允许先保存，后续业务办理前必须可识别并补全的字段。

当前落地：

- 新增 `sc.system.default.mixin`，提供 `含系统默认值`、`系统默认字段`、`系统默认说明` 三个标识字段。
- 物资链路模型接入系统默认兜底：采购申请、材料进场验收、入库单、出库单、询比价、材料结算及其明细。
- 系统默认值仅用于草稿保存和级联创建不失败，列表/表单会提示该记录含系统默认值。

当前默认记录语义：

- `系统默认项目（待完善）`
- `系统默认供应商（待完善）`
- `系统默认材料（待完善）`
- `系统默认仓库（待完善）`

后续要求：

- 任何新增正式业务模型，如果存在 required 字段但可能在级联创建、历史重放、快速创建中缺失，必须接入同类策略。
- 不能把系统默认值当成真实业务事实参与最终审批或结算，应在关键业务动作前提示审批人或经办人补全。

## 已闭环

### 费用中心菜单

反馈：费用中心需要包含公司扣款、公司收入、公司支出、备用金、借款单、还款单、费用报销单、项目费用报销单。

状态：已补菜单承载入口，归入财务/费用中心业务分类。

提交：`1fbf2464 Expand feedback-driven business fact coverage`

### 供应商

反馈：

- 供应商菜单需要进入基础设置。
- 列表字段存在英文名，如 `Complete Name`、`Job Position`。
- 需要增加供应商类型。
- 表单字段需要账户信息、账户名称、开户银行、账号、备注、附件、统一社会信用代码、所属地区。
- 历史供应商数量与旧系统有差异，需要后续数据重放核对。

状态：已补供应商账户/信用/地区等事实字段和供应商菜单/视图承载。历史数量差异待模拟生产库重放核对。

提交：`1fbf2464 Expand feedback-driven business fact coverage`

### 合同体系边界

反馈：收入合同、支出合同、一般合同、历史合同台账混在一起，字段和菜单定位不清。

状态：

- `construction.contract` 定位为项目收入/支出合同正式模型。
- `sc.general.contract` 定位为公司一般合同。
- `sc.legacy.purchase.contract.fact` 降级为历史事实承载，不再作为新业务审批对象。
- 合同审批范围、动作、序列、菜单和视图已随模型边界调整。

提交：`0a64a090 Normalize contract fact model boundaries`

### 合同字段口径

反馈：

- “合同明细金额”应按旧系统语义改为最终合同价。
- 收入合同台账需展示合同编号、结算金额、开票金额、未开票金额、收款金额、未收款金额、挂靠人、工程地址。
- 支出合同台账需展示结算金额、是否归档、已付款金额、未付款金额、已开票金额、未开票金额。
- 列表字段“合同编号”应表达为“单据编号”。
- 需要录入人、录入数据、经办人等历史承接字段。

状态：正式合同和一般合同已补执行金额、录入/经办、工程地址、归档等字段，并调整核心列表/表单视图。部分字段是否能从旧库全量回填，需在模拟生产库重放后验算。

提交：

- `0a64a090 Normalize contract fact model boundaries`
- `1fbf2464 Expand feedback-driven business fact coverage`

### 审批风险语义

反馈：业务规则不满足也应允许审批人判断并同意推进，系统不能因为风险提示阻断业务。

状态：付款申请审批回调已从硬阻断改为风险提示；结算状态、超付、资金基线等风险进入 chatter 提示，审批人仍可通过。保留项目关闭、结算取消等数据闭环硬约束。

提交：`6ae2e44e Align approval gates with advisory review`

### 材料计划

## 待闭环

### 历史数据承接核对

问题：

- 供应商旧系统截图显示约 5111 条，新系统当前统计不一致。
- 收入/支出合同列表的历史数据存在字段未展示或未回填情况。
- 合同结算、入库单等历史事实是否已完整进入正式模型还需按模拟生产库验证。

下一步：

1. 在模拟生产库执行一键重建或升级。
2. 对供应商、收入合同、支出合同、一般合同、合同结算、材料计划、询比价、入库单做数量和关键字段抽样。
3. 输出字段映射缺口，区分“模型缺字段”和“重放脚本未映射”。
4. 对缺口补迁移/重放映射，确保全新重建不得复发。

### 收入合同结算

反馈：

- 表单需增加单据日期、发包人、承包人、标题、送审金额、审定金额、审定日期、项目名称、合同名称、合同总额、申请资金金额、工程地址等旧系统字段。
- 列表需增加标题、结算单位、扣款金额、已付款金额、未付款金额、结算说明、附件、录入人。

判断：这部分更像合同结算/结算单正式模型能力，不应放到历史截图字段里临时展示。需要确认当前 `sc.settlement.order` 是否已有对应字段或需要扩展。

迭代步骤：

1. 盘点 `sc.settlement.order` 与旧系统合同结算字段映射。
2. 补模型字段：标题、结算单位、扣款金额、结算说明、录入人、附件等。
3. 将已付款/未付款按现有付款事实计算，不手填。
4. 更新收入/支出结算列表和表单。
5. 增加升级测试和历史数据重放断言。

### 一般合同

反馈：

- 菜单“支出合同/一般合同”需明确为“一般合同（公司）”。
- 列表和表单需增加联系人、联系电话。

状态：模型已具备联系人/联系电话字段，列表/表单已部分展示。菜单命名仍需二次核对是否所有入口都统一为“一般合同（公司）”。

迭代步骤：

1. 搜索所有一般合同菜单和 action 名称。
2. 统一文案为“一般合同（公司）”。
3. 确认列表默认列含联系人、联系电话。
4. 补视图测试。

### 询比价

反馈：

- 表单需增加联系人、联系电话、备注。
- 同一询价需要支持多家供应商同时展现。

判断：联系人/电话/备注是询价主表事实；多供应商是询价对象明细或报价明细能力，不能只用单一 `partner_id` 表达。

迭代步骤：

1. 盘点询比价模型是否已有供应商明细表。
2. 若无，增加供应商明细模型或 One2many 字段，承载供应商、联系人、电话、报价状态、备注。
3. 表单展示多供应商明细。
4. 列表增加联系人/联系电话等关键字段。
5. 补创建、读取、升级测试。

### 入库单

反馈：

- 明细需增加单价、金额、合计、备注。
- 表单需增加备注。
- 填单后保存失败，且返回后已填数据丢失。

判断：这是当前最影响可用性的阻断项。需要先定位保存失败原因，再补字段。保存失败不能靠前端吞掉错误，必须让后端模型约束和前端错误提示一致。

当前进展：

- 入库单已接入模型层默认兜底：缺项目、仓库、库位时自动填入系统默认并打标识。
- 入库明细缺材料、数量时自动填入系统默认材料和数量 1，并打标识。
- 明细已具备单价、金额、备注，表单列表已展示系统默认标识。

迭代步骤：

1. 浏览器复现入库单创建失败，记录请求和后端 traceback。
2. 检查入库单必填字段、默认值、onchange、权限和状态转换。
3. 修复保存失败，确保失败时前端保留已填草稿状态。
4. 增加入库明细单价、金额、合计、备注字段。
5. 更新表单/列表/明细视图。
6. 增加入库单创建 smoke 测试。
7. 在模拟生产库升级后验证：用户不填仓库/库位/明细材料时可以保存草稿，且记录明确显示系统默认待完善。

### 材料进场验收/采购链路

反馈：材料计划、采购申请、材料进场验收、采购订单、询比价、入库单之间需要更完整的字段承接。

状态：材料计划和材料进场验收已补一轮字段；采购申请/采购订单/询比价/入库单还需要按旧系统截图继续核对。

迭代步骤：

1. 以“材料计划 -> 询比价 -> 采购订单 -> 进场验收 -> 入库单”为主链路跑通。
2. 每个节点确认来源单据、供应商、材料、规格、单位、数量、金额、附件、备注是否传递。
3. 对不能直接传递的字段，明确是引用、复制还是计算。
4. 补链路测试。

## 推荐执行顺序

1. 先修复入库单保存失败。该问题直接影响用户填单，是当前最高优先级。
2. 补入库单金额字段和备注字段。
3. 补合同结算字段和视图。
4. 补询比价多供应商能力。
5. 统一一般合同菜单和联系人/电话列表列。
6. 在模拟生产库做一键重建，核对历史数据数量和字段映射。
7. 补测试并分类提交。

## 验收清单

- 供应商、合同、结算、材料计划、询比价、入库单页面列表字段均为中文业务字段。
- 历史数据重放后关键模型数量与旧系统口径可解释一致。
- 表单创建失败时有明确错误，且不丢失用户已填内容。
- 审批业务风险只提示不阻断，审批人可以继续推进。
- 数据闭环硬约束仍生效，例如项目关闭前未完成结算/付款不可直接关闭。
- 全新重建和模块升级后字段、视图、权限、菜单均自动生效。
