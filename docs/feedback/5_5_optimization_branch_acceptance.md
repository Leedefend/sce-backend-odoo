# 5.5 用户反馈分支迭代目标与视觉验收

来源文件：`/home/odoo/workspace/5.5优化.docx`

分支：`feedback/5-5-optimization`

本文档是本分支唯一的用户反馈闭环台账。后续推进以“用户打开页面能否看到并完成”为准，不以“代码里有字段/方法”为准。每一项必须同时给出用户视觉验收方式、当前状态和证据。

## 验收原则

- 用户视觉优先：以浏览器页面、列表列头、表单字段、保存后的跳转、筛选/分组结果为验收对象。
- 模拟生产优先：涉及 Odoo 模型、字段、视图、菜单的改动，必须先在 `sc_prod_sim` 完成模块升级。
- 业务路径优先：仅静态检查、前端构建、字段存在不算闭环；必须验证用户能创建、保存、筛选或从来源单带入。
- 不扩大解释：截图中的“所有菜单”“所有财务菜单”按可见业务中心逐项盘点，不用几个核心模型代替全量菜单验收。
- 未经浏览器截图或录屏确认的项，只能标为“后端已生效”或“待视觉验收”，不能标为“已闭环”。

## 本分支必须补齐的剩余范围

### A. 全量财务菜单盘点

目标：不能只改截图中最明显的几个模型。必须从财务中心菜单树列出所有可新增业务单据，逐项判断是否需要单据日期、备注、附件、收款账户信息、付款账户信息。

验收：

- 输出一张“财务菜单字段覆盖表”。
- 每个财务表单至少给出“已覆盖 / 不适用 / 待补齐”的结论。
- 用户打开财务中心逐项新建时，字段不会时有时无。

### B. 当前项目上下文全菜单默认策略

目标：用户选择当前项目后，不应在项目型业务单据里重复选项目。

验收：

- 输出一张“项目型菜单默认项目覆盖表”。
- 项目型单据新建时项目自动带入。
- 列表视图至少能按当前项目聚焦或一键筛选。
- 非项目型单据明确标记“不适用”。

### C. 浏览器视觉验收

目标：用用户视角确认，不再只看后端断言。

最低截图证据：

- 项目立项快速创建：显示单据日期，保存后返回列表/入口。
- 投标表单：当前项目、业主自动带入，空清单保存不报错。
- 投标保证金表单：单据日期、收款账户、付款账户、备注、附件可见。
- 收入合同列表：税额、最终合同价、经营策略可见。
- 付款申请表单：单据日期、备注、附件、经营策略可见，能生成付款登记。
- 付款执行表单：选择付款申请后自动带入数据，收/付款账户结构化字段可见。
- 收款收入表单：账户名称、账号、开户行、备注、附件、经营策略可见。

## 状态定义

- 未开始：尚未定位模型/视图/菜单。
- 处理中：已有改动但未升级或未验证。
- 后端已生效：模拟生产升级和 Odoo shell 断言通过。
- 待视觉验收：后端或前端验证通过，但尚无浏览器截图证据。
- 已闭环：模拟生产升级通过、业务断言通过、浏览器视觉验收通过。
- 不适用：该反馈不适用于该模型或菜单，已给出原因。

## 当前结论

本分支目前不是“用户反馈全部闭环”状态。准确状态是：

- 核心后端业务路径已在模拟生产生效。
- 前端构建和类型检查通过。
- 浏览器视觉验收尚未执行。
- “所有财务菜单”和“所有项目型菜单”两个泛化反馈尚未全量覆盖。
