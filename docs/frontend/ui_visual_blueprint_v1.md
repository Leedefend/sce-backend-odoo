# Frontend Visual Blueprint v1

## 1. Document Position

本文件用于把 `frontend/apps/web` 的前端视觉方向从“组件收口说明”提升为“产品见面级蓝图”。

本轮定位：

- Layer Target: `Frontend contract-consumer runtime`
- Module: `docs/frontend visual blueprint`
- Scope: 产品气质、视觉哲学、页面母版、组件优先级、后续批次建议

硬约束：

- 前端继续作为 Odoo 原生语义与动态契约的通用消费者
- 不以前端特判修补后端语义缺口
- 蓝图服务于共享 shell、共享页面承接层与后续 UI-check 批次

## 2. 产品视觉总定位

产品定位统一定义为：

**新一代企业级项目管理工作台**

它既不是传统 ERP 的拥挤陈旧后台，也不是轻飘娱乐化的互联网工具。目标气质固定为：

- 专业可信：状态清楚、信息有序、操作稳定
- 现代克制：通过比例、留白、层级和一致性建立时代感
- 管理者友好：支持负责人、项目经理、审批与财务角色快速判断
- 智能感可感知：风险提示、推荐动作、下一步建议拥有独立表达层

## 3. 设计哲学

后续页面和组件必须服从以下母原则：

1. 信息优先于装饰
2. 统一优先于局部创意
3. 业务状态必须成为视觉语言
4. 骨架稳定，内容可变
5. 专业感来自克制

禁止的设计漂移：

- 色彩过多
- 阴影过重
- 圆角过大
- 动效过密
- 卡片过碎
- 图标来源混杂

## 4. 用户感知目标

用户首次进入系统时，应形成以下连续感知：

- 第一感觉：系统很稳，不乱
- 第二感觉：状态清楚，重点易找
- 第三感觉：明显比传统后台现代，但不轻浮
- 第四感觉：像一个成熟产品，而不是拼装页面集合

## 5. 行业与品牌表达

工程项目管理的行业属性通过信息优先级体现，而不是通过装饰图案体现。页面要自然露出以下特征：

- 计划性
- 执行力
- 责任边界
- 过程控制
- 风险预警
- 结果闭环

因此整体风格不能走：

- 电商后台
- CRM 营销风
- BI 大屏炫彩风
- 轻社交工具风

## 6. Visual Tokens Direction

现有 token 体系继续沿用 `tokens.css` 的技术底盘，产品语义方向补充如下。

### 6.1 Color Strategy

- 主品牌色：稳定、理性、干净的蓝色系，用于主操作、导航选中与关键链接
- 中性色：作为主角，控制背景、边框、文本层级和结构分隔
- 状态色：灰/蓝/绿/橙/红固定映射草稿、进行中、完成、预警、风险
- 智能语义色：蓝青或蓝灰，用于 AI 建议、提示与辅助分析

### 6.2 Typography Strategy

- 页面标题建立权威感
- 区块标题承担分区
- 正文保证高密度下可读
- 辅助文字主动后退
- 关键数字必须更有力量

### 6.3 Spacing and Shape

- 所有页边距、区块间距、卡片内边距由 spacing token 驱动
- 圆角保持轻圆角，不做卡通化处理
- 结构组织优先依赖边框与弱层次表面，不依赖重阴影

## 7. Page Skeleton Baseline

全系统页面统一采用三段式壳层：

1. 左侧导航区
2. 顶部全局条
3. 中央主工作区

主工作区内部统一为：

1. 页面定位区
2. 主标题与操作区
3. 筛选或视图控制区
4. 核心内容区
5. 辅助内容区

这个秩序必须先于业务内容成立。

## 8. 页面母版

后续页面统一从四类母版生长：

1. 首页/工作台页
2. 列表页
3. 详情/表单页
4. 驾驶舱/总览页

统一的东西：

- 顶部定位方式
- 内容容器宽度
- 区块间距
- 标题层级
- 动作区位置
- 状态表达语言

允许变化的东西：

- 业务区块内容
- 数据密度
- 辅助信息占比
- 场景特化组件

## 9. 首页与工作台母版

首页不是装饰型门户，而是第一工作界面。

标准结构：

1. 欢迎与角色工作摘要
2. 核心待办与推荐动作
3. 高频业务入口
4. 项目动态与风险提醒

工作台页比首页更强调角色场景，围绕以下事实组织：

- 我的事项
- 我的责任
- 我的进度
- 我的异常
- 我的动作

推荐布局：

- 顶部单列摘要
- 中部双列，左宽右窄
- 下部网格承载入口与补充信息

## 10. 列表页母版

列表页统一为六段式：

1. `PageHeader`
2. 工具动作区
3. 筛选与搜索区
4. 列表摘要区
5. 数据表格区
6. 分页与总量区

列表页的设计目标固定为：

- 找
- 筛
- 比
- 进入详情
- 发起批量处理

信息权重规则：

- 主字段更醒目，可带副标题
- 次字段清楚但不抢主字段
- 辅助字段主动后退，不打断浏览节奏

## 11. 详情与表单页母版

详情页不是字段堆栈，而是业务对象控制台。

标准结构：

1. `PageHeader`
2. 概览区
3. 主体信息区
4. 关联信息区
5. 活动与日志区

统一分区建议：

- 基本信息
- 业务信息
- 执行信息
- 财务信息
- 附属信息

关键要求：

- 先建立对象全局理解，再下沉字段
- 当前状态、负责人、阶段、风险和下一步动作优先进入视线
- `notebook/tab`、section、chatter 都应服从同一视觉秩序

## 12. 驾驶舱母版

驾驶舱用于快速判断，而不是堆数据。

建议结构：

1. 标题与周期区
2. 核心指标区
3. 风险与待办区
4. 趋势与分析区
5. 明细入口区

优先回答：

- 当前是否正常
- 风险在哪
- 哪个指标最值得看
- 下一步做什么

## 13. Global Structural Components

### 13.1 Sidebar

Sidebar 承担：

- 系统结构地图
- 当前定位器
- 高频入口集合
- 模块边界展示器

视觉要求：

- 一级菜单模块感明确
- 二级菜单更轻但层次清晰
- 选中态必须强于 hover
- 图标统一线性风格
- 菜单密度避免过高

### 13.2 Topbar

Topbar 建立全局控制感，不堆功能。

建议承载：

- 上层定位
- 用户入口
- 全局搜索预留
- 通知与快捷入口预留

### 13.3 PageHeader

统一结构：

- 左侧：标题、副标题、状态
- 右侧：主动作、次动作、更多动作
- 可选下层：统计、标签、业务摘要

## 14. Shared Component Priorities

组件系统按以下优先级持续收口：

1. `Button`
2. `Input`
3. `StatusTag/Badge`
4. `Card`
5. `Table`
6. `PageFeedback`
7. 空态/加载态/错误态/无权限态

设计要求：

- 主次动作必须区分
- 状态组件必须全系统统一
- 风险组件不能退化为一行红字
- AI 建议组件要独立但克制

## 15. Business Visual Semantics

业务视觉语义层是产品差异点，必须固定三类表达：

- 状态语义：项目、任务、审批、财务对象进入统一状态语言
- 风险语义：通过 tag、card、summary 和动作提示形成完整表达
- 动作语义：主动作、次动作、危险动作分层清晰

同时保留智能语义层，用于：

- AI 建议
- 推荐动作
- 分析提示
- 风险解释

## 16. Relationship With Existing Runtime Baseline

本蓝图不替代现有实现文档，而是作为其上位设计依据：

- `docs/frontend/ui_style_system_convergence_v1.md` 记录已经落地的 token、shell、shared primitives 和页面承接层收口
- 本文档定义这些收口动作应朝向的产品气质与页面秩序
- 后续 UI-check、Playwright、page-level convergence 批次应同时引用两份文档

## 17. Current Gaps Against This Blueprint

相对本蓝图，当前仓库仍待收口的区域包括：

- 项目/任务详情页的 notebook、section、chatter 辅助区
- 表格更细粒度的行态、排序态、操作列规范
- 首页/工作台与驾驶舱的统一母版
- 浏览器内 before/after 截图与人工 walkthrough 证据

## 18. 后续批次建议

建议顺序：

1. 开 UI-check / Playwright 批次，验证项目列表、任务列表、项目详情、任务详情与本蓝图一致性
2. 开详情页辅助区收口批次，统一 notebook/section/chatter 的视觉秩序
3. 开首页/工作台母版批次，把待办、风险、推荐动作落到稳定模板
4. 逐步将页面验收从“工程门禁 PASS”提升到“浏览器内产品观感 PASS”

## 19. Final Statement

本轮前端视觉方向最终统一为：

**稳重专业的工程项目管理工作台，拥有现代企业软件的清爽秩序感，并通过统一状态语言与智能提示层建立区别于传统 ERP 的时代感。**
