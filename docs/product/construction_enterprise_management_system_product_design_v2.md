# 施工企业管理系统产品设计文档 v2（可实施版）

## 1. 产品目标与定位

### 1.1 产品目标

构建一套以项目为核心驱动的施工企业管理系统，实现：

- 项目全过程闭环管理
- 多业务域统一承载
- 系统自动引导业务推进
- 支持逐步扩展为平台型系统

### 1.2 产品核心能力

```text
项目驱动 + 状态驱动 + 行为驱动
```

### 1.3 产品定位

| 类型 | 定位 |
| --- | --- |
| v2 | 可运行产品 |
| v3 | 可扩展平台 |
| 长期 | 多行业系统 |

---

## 2. 产品总体结构

### 2.1 三层结构

系统总体结构分为三层：

```text
① 主线（项目闭环）
② 能力域（业务承载）
③ 功能点（用户需求）
```

### 2.2 项目主线

项目主线是整个系统的骨架：

```text
立项 -> 计划 -> 执行 -> 结算 -> 完成
```

说明：

- 成本和付款属于执行期业务域
- 不单独升格为项目主阶段

### 2.3 能力域结构

系统能力域分为核心能力域和支撑能力域。

#### 2.3.1 核心能力域

覆盖：

## 3. 项目模型设计

### 3.1 主字段

```python
project.project

stage
milestone
progress
cost_total
payment_total
risk_level
```

### 3.2 阶段模型

```text
draft
initiated
planned
executing
settled
done
```

### 3.3 里程碑模型

```text
planning_ready
execution_started
execution_mid
execution_near_complete
cost_baselined
payment_started
payment_completed
ready_for_settlement
```

### 3.4 三层关系

| 层 | 作用 |
| --- | --- |
| `stage` | 主流程骨架 |
| `milestone` | 中粒度推进 |
| `data` | 业务事实层 |

### 3.5 模型原则

- `stage` 表达主经营阶段
- `milestone` 表达阶段内推进深度
- `data` 表达真实业务记录
- 系统引导逻辑必须基于 `stage + milestone + data`

## 4. 业务主流程

系统必须至少保证以下主路径可走通：

```text
创建项目
-> 确认立项
-> 创建任务
-> 开始执行
-> 更新任务
-> 录入成本
-> 发起付款
-> 进入结算
-> 完成项目
```

该主流程是最小经营闭环，不代表后续所有复杂业务都必须在 v2 中一次性实现。

## 5. 核心系统设计

### 5.1 项目驾驶舱

项目驾驶舱是项目经营闭环的统一入口，不是普通菜单页。

#### 页面结构

##### 1. 项目头部

- 项目名称
- 负责人
- 当前阶段

##### 2. 当前阶段与下一步

示例：

```text
当前阶段：执行中

下一步：
-> 录入成本
-> 更新任务
```

##### 3. 进度区

- 任务完成率
- 延期风险

##### 4. 成本资金区

- 成本
- 已付款
- 待付款

##### 5. 风险区

- 风险等级
- 风险提示

##### 6. 快捷入口区

- 关键能力入口
- 当前角色最常用动作

#### 驾驶舱原则

- 必须展示当前阶段
- 必须展示下一步
- 用户无需猜测下一步去哪做

### 5.2 next_actions 引擎

#### 数据结构

```json
{
  "key": "",
  "label": "",
  "target_scene": "",
  "action_kind": "guidance/transition",
  "condition": "",
  "priority": 1
}
```

#### 规则

- `guidance`：引导处理，不推进主阶段
- `transition`：阶段性动作，允许推进主阶段

#### 约束

- 所有项目主阶段都必须有 next_actions
- next_actions 应同时参考 `stage`、`milestone` 和必要业务事实
- 页面展示动作不等于状态推进动作

### 5.3 阶段推进机制

允许的阶段性动作：

```text
confirm_initiation
confirm_plan_ready
start_execution
confirm_settlement
complete_project
```

禁止事项：

```text
直接修改 stage
```

## 7. 功能覆盖原则

### 7.1 总原则

所有用户功能都必须存在，但可以重构位置、承载方式和入口组织。

### 7.2 映射规则

系统功能统一通过以下路径落位：

```text
功能 -> 能力域 -> 阶段 -> 场景
```

### 7.3 设计约束

- 不允许用户功能成为孤立页面
- 不允许能力域脱离项目主线长期漂浮
- 不允许把菜单结构误写成业务模型

## 9. 系统约束

### 9.1 约束一

禁止直接修改 `stage`。

### 9.2 约束二

必须通过业务行为推进主阶段。

### 9.3 约束三

系统必须始终提供 next_actions。

### 9.4 约束四

系统必须始终能给用户明确引导。

### 9.5 约束五

能力域扩展不得破坏项目主线骨架。

---

## 10. 验收标准

### 10.1 用户验收问题

用一个真实用户场景判断系统是否成立：

- 能否创建项目
- 能否推进项目
- 每一步是否有引导
- 能否完成项目

### 10.2 判定原则

如果用户在主路径中的关键环节卡住，或系统无法说明“下一步”，则本产品设计不成立。

---

## 11. 最终结论

本设计文档的目标不是堆积功能，而是建立一套：

- 覆盖用户需求
- 保持业务闭环
- 支持分阶段实施
- 可继续平台化扩展

的施工企业管理系统产品基线。

它是系统的产品总体蓝图，也是后续各阶段开发、切片规划和验收的统一依据。
