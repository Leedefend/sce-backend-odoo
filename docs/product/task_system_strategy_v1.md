# 任务体系策略 v1

状态：`Frozen`

## 核心结论

项目上的所有可执行事项，底层都属于任务域。

任务体系不等于“用户手工创建的 `project.task`”，而应定义为：

`显性任务 + 系统派生任务`

## 一、两类任务

### 1. 显性任务

显性任务以 Odoo 原生 `project.task` 为核心载体。

适用场景：

- 用户主动创建执行任务
- 需要明确排期、指派、协同
- 需要沿用 Odoo 原生任务能力

执行原则：

- 优先复用 Odoo 原生任务模型与视图
- 后端治理层只做用户表面裁剪
- 不重建第二套任务主数据

### 2. 系统派生任务

系统派生任务指由业务对象自动映射出来的执行事项。

典型来源：

- 合同录入
- 成本登记
- 付款申请
- 结算处理
- 风险处置
- 资料补录
- 里程碑推进

执行原则：

- 用户不需要先手工定义任务
- 系统应自动把这些业务动作纳入任务管理视角
- 当前阶段允许先做“任务投影”，不强制全部物化为 `project.task`

## 二、当前阶段架构口径

### 真源

- 显性任务真源：`project.task`
- 业务真源：合同/成本/付款/结算/证据/风险等业务对象

### 统一层

后端建立统一任务视角：

- `Task Projection`
- `Task Unification`

用于把显性任务和系统派生任务汇总成统一执行面。

### 前端口径

前端消费统一任务视角，不要求用户重复维护：

- 业务对象一份
- 任务对象再一份

## 三、产品化原则

### 1. 充分借用 Odoo 原生任务体系

- 继续使用 `project.task`
- 继续借用原生任务表单、列表、看板、阶段、指派等能力
- 优先走“原生视图治理 + 前端消费”

### 2. 不要求用户显式建任务才算进入任务系统

业务动作一旦具备：

- 执行主体
- 状态
- 时限
- 下一步动作

就应能进入任务管理体系。

### 3. 区分“物化任务”和“投影任务”

当前阶段不要求所有业务事项立即物化成 `project.task`。

允许两种方式并存：

- 物化：真正创建 `project.task`
- 投影：由后端统一投影进任务视图

## 四、当前阶段实施顺序

### 第一阶段

- 继续治理 `project.task` 标准 list/form
- 确保显性任务主链可用

### 第二阶段

- 把合同、成本、付款、结算等高频业务对象纳入统一任务投影
- 在“我的待办 / 项目执行 / 项目驾驶舱”中可见

### 第三阶段

- 再决定哪些系统派生任务需要真正物化成 `project.task`
- 哪些只保留投影视图即可

## 五、当前冻结口径

一句话定义：

> 任务体系 = Odoo 原生 `project.task` + 业务对象自动派生的执行事项。

一句话执行原则：

> 用户可以显式创建任务，但不是必须；系统应自动把项目业务执行纳入任务管理体系。
