# 施工企业项目管理业务闭环规范 v1.1（草案）

## 1. 目标与范围

### 1.1 目标

定义施工企业项目从立项到结算的最小业务闭环，使系统具备以下能力：

- 可连续操作
- 可状态驱动
- 可由系统自动引导下一步
- 可作为后续开发、验收与产品演进基线

本规范的目标不是描述一个完整 ERP，而是定义一个可跑通、可收口、可持续扩展的施工企业项目经营闭环。

### 1.2 范围

本规范覆盖以下主业务链：

```text
项目立项 -> 项目计划 -> 项目执行 -> 成本记录 -> 付款记录 -> 项目结算
```

当前不包含：

- 招投标
- 合同深度条款管理
- 财务核算与总账
- 发票/税务
- 多行业通用建模

### 1.3 规范定位

本规范是产品规范草案，不等同于代码现状快照，也不直接替代已冻结发布文档。

它的作用是：

- 为后续产品设计提供统一口径
- 为实现演进提供业务约束
- 为开发时判断“是否符合闭环逻辑”提供依据

---

## 2. 业务主流程

施工企业项目的目标主线定义为：

```text
[立项]
   ↓
[计划]
   ↓
[执行]
   ↓
[结算]
   ↓
[完成]
```

其中：

- 成本记录与付款记录是执行期内持续发生的业务域，不单独升格为项目主阶段
- 结算是执行期后半段的收口动作

### 2.1 主流程原则

- 主流程单向推进
- 允许在执行期内回补成本、付款等业务数据
- 不允许跨阶段跳跃式推进主阶段
- 不允许通过直接改字段推进项目状态

---

## 3. 项目阶段模型

### 3.1 核心字段

```text
project.stage
```

### 3.2 状态枚举

```text
draft       （草稿）
initiated   （已立项）
planned     （已计划）
executing   （执行中）
settled     （已结算）
done        （已完成）
```

### 3.3 状态流转

```text
draft -> initiated -> planned -> executing -> settled -> done
```

### 3.4 强约束

- 只能顺序推进
- 必须由显式业务行为驱动推进
- 禁止手工直接修改 `project.stage`
- 列表查看、详情查看、汇总查看等读取行为不得推进主阶段

### 3.5 说明

以下业务域不作为主阶段字段值：

- 成本处理中
- 付款处理中

原因是：

- 成本和付款在施工项目中通常是执行期内持续发生的经营活动
- 若把它们升为主阶段，会导致项目阶段频繁摆动，破坏主流程语义

### 3.6 项目里程碑模型

除主阶段外，项目还需要一层中粒度业务推进表达：

```text
project.milestone
```

该字段用于表达：

- 当前执行深度
- 是否已进入收尾
- 是否具备结算前置条件
- next_actions 的精细驱动条件

### 3.7 里程碑枚举

建议最小枚举如下：

```text
planning_ready
execution_started
execution_mid
execution_near_complete
cost_baselined
payment_started
payment_completed
ready_for_settlement
```

### 3.8 里程碑约束

- `milestone` 不替代 `stage`
- `milestone` 不应逆向改写主阶段
- `milestone` 用于细粒度推进表达、驾驶舱提示和 next_actions 生成
- 允许在同一主阶段内推进多个里程碑

### 3.9 stage / milestone / data 三层关系

| 层级 | 作用 | 说明 |
| --- | --- | --- |
| `stage` | 主流程骨架 | 控制项目处于哪一个经营大阶段 |
| `milestone` | 中粒度推进 | 控制当前阶段内推进到什么程度 |
| `data` | 业务事实层 | 任务、成本、付款、结算等具体记录 |

系统设计原则是：

- `stage` 负责骨架
- `milestone` 负责推进
- `next_actions` 负责引导
- `data` 负责提供事实依据

## 4. 阶段定义

### 4.1 草稿阶段（draft）

#### 进入条件

- 新建项目草稿

#### 退出条件

- 完成项目基本信息创建

#### 允许的阶段性动作

- `confirm_initiation`

#### 写回

```text
project.stage = initiated
```

### 4.2 立项阶段（initiated）

#### 进入条件

- 项目已创建并完成基本立项信息

#### 必做事项

- 明确项目名称
- 明确负责人
- 明确基础归属信息

#### 退出条件

- 项目被确认进入计划准备

#### 允许的阶段性动作

- `confirm_plan_ready`

#### 写回

```text
project.stage = planned
```

### 4.3 计划阶段（planned）

#### 进入条件

- 项目已立项

#### 必做事项

- 创建任务结构
- 形成最小计划骨架

#### 退出条件

- 至少存在一组可执行任务

#### 允许的阶段性动作

- `start_execution`

#### 写回

```text
project.stage = executing
```

### 4.4 执行阶段（executing）

#### 进入条件

- 已存在任务结构并进入执行

#### 必做事项

- 更新任务执行状态
- 录入执行过程数据
- 录入成本
- 发起和跟踪付款

#### 关键里程碑

- `execution_started`
- `execution_mid`
- `execution_near_complete`
- `cost_baselined`
- `payment_started`
- `payment_completed`
- `ready_for_settlement`

#### 退出条件

- `project.milestone = ready_for_settlement`

#### 结算前置条件（最小口径）

达到 `ready_for_settlement` 前，至少应满足：

- 项目已进入执行后段
- 已存在成本记录
- 已存在付款记录
- 驾驶舱能够给出结算收口提示

本版仍采用最小业务口径，不在此处展开复杂财务或合同规则。

#### 允许的阶段性动作

- `confirm_settlement`

#### 写回

```text
project.stage = settled
```

### 4.5 结算阶段（settled）

#### 进入条件

- 项目已完成结算确认

#### 必做事项

- 固化结算结果
- 确认项目经营收口

#### 关键里程碑

- `ready_for_settlement`

#### 退出条件

- 项目确认完结

#### 允许的阶段性动作

- `complete_project`

#### 写回

```text
project.stage = done
```

### 4.6 完成阶段（done）

#### 进入条件

- 项目已完成业务收口

#### 状态说明

- 该阶段为闭环终态
- 允许查看历史，不应再继续推进

## 5. 业务阶段与发布切片的关系

### 5.1 原则

业务阶段不等于发布切片。

业务阶段描述的是项目经营推进状态。  
发布切片描述的是系统当前对外提供的产品入口与能力切面。

二者必须关联，但不能简单做一一同名映射。

### 5.2 当前已发布切片

当前已冻结发布面为：

- FR-1 项目立项
- FR-2 项目推进
- FR-3 成本记录
- FR-4 付款记录
- FR-5 结算结果

### 5.3 承载关系

| 发布切片 | 主要承载内容 | 业务阶段关系 |
| --- | --- | --- |
| FR-1 项目立项 | 项目创建、立项入口 | `draft -> initiated` |
| FR-2 项目推进 | 驾驶舱、计划、执行主链 | `initiated -> planned -> executing` |
| FR-3 成本记录 | 执行期成本录入与汇总 | 执行期业务域，不单独推进主阶段 |
| FR-4 付款记录 | 执行期付款录入与汇总 | 执行期业务域，不单独推进主阶段 |
| FR-5 结算结果 | 项目结算只读汇总 | `executing -> settled` 的收口承载 |

### 5.4 禁止事项

- 禁止把“FR-3 = costing 主阶段”“FR-4 = paying 主阶段”直接写入主阶段模型
- 禁止因为切片存在就强制主阶段同步变为同名状态

---

## 6. next_actions 标准

### 6.1 设计原则

系统必须在任何时候给用户一个明确的下一步，但“下一步”分两类：

- 引导动作：帮助用户继续业务处理
- 阶段性动作：真正会推进 `project.stage`

### 6.2 统一结构

```json
{
  "key": "cost.enter",
  "label": "录入成本",
  "target_scene": "cost.tracking.enter",
  "condition": "project.stage == executing && project.milestone >= execution_started",
  "priority": 1,
  "action_kind": "guidance"
}
```

阶段推进动作示例：

```json
{
  "key": "project.start_execution",
  "label": "开始执行",
  "target_scene": "project.execution.enter",
  "condition": "project.stage == planned && project.milestone == planning_ready",
  "priority": 1,
  "action_kind": "transition"
}
```

### 6.3 强约束

- `guidance` 动作不得直接推进 `project.stage`
- `transition` 动作必须有明确业务前置条件
- `next_actions` 应同时结合 `stage` 与 `milestone` 生成，而不是只依赖主阶段
- 页面展示动作不等于阶段推进动作

### 6.4 各阶段建议动作

#### initiated

- 进入项目推进
- 开始计划准备

#### planned

- 创建任务结构
- 开始执行

#### executing

当 `milestone = execution_started`：

- 更新任务
- 录入成本

当 `milestone = execution_mid`：

- 继续更新任务
- 补充成本
- 发起付款

当 `milestone = execution_near_complete`：

- 查看付款状态
- 补全收尾数据
- 评估是否进入结算准备

当 `milestone = ready_for_settlement`：

- 进入结算
- 确认结算

#### settled

- 查看结算结果
- 完成项目

#### done

- 查看项目总结

## 7. 项目驾驶舱

项目驾驶舱是业务闭环的核心入口，不是普通导航页。

### 7.1 必须展示

- 当前主阶段
- 当前里程碑
- 项目进展摘要
- 成本汇总
- 付款汇总/状态
- 结算摘要（若已具备）
- 风险提示
- 下一步操作

### 7.2 设计原则

- 用户无需思考“该去哪一步”
- 用户打开驾驶舱后，必须能看到当前状态和下一步
- 驾驶舱优先服务行动，而不是展示技术信息
- 驾驶舱应优先展示“当前阶段 + 当前里程碑 + 下一步”

---

## 8. 用户标准操作路径

### 8.1 标准路径

```text
进入系统
-> 创建项目
-> 确认立项
-> 进入项目推进
-> 创建任务
-> 开始执行
-> 更新任务
-> 录入成本
-> 发起付款
-> 查看结算结果
-> 完成项目
```

### 8.2 路径原则

- 每一步必须有入口
- 每一步必须有下一步
- 系统应减少用户自行推理流程的成本

---

## 9. 数据关系（简化版）

```text
项目（project）
├── 任务（task）
├── 成本（cost）
├── 付款（payment）
└── 结算（settlement）
```

### 9.1 关系原则

- 项目是经营闭环锚点
- 成本、付款、结算均以项目为归属
- 不允许脱离项目上下文定义成本、付款与结算闭环

---

## 10. 系统约束

### 10.1 约束一

禁止手工直接修改 `project.stage`。

### 10.2 约束二

只有明确的阶段性业务动作才能写回 `project.stage`。

### 10.3 约束三

读取类页面和读取类动作不得写回 `project.stage`。

### 10.4 约束四

每个主阶段都必须提供 `next_actions`。

### 10.5 约束五

系统必须保证任何时候都能给用户提供“当前在哪”和“下一步做什么”。

---

## 11. 验收标准

以下问题必须能被一个真实用户连续回答为“是”：

- 能否创建项目
- 能否一步步推进项目
- 每一步是否都知道该做什么
- 执行期内是否能完成成本与付款处理
- 最终是否能完成项目结算与项目完结

如果用户在任一关键阶段卡住，或系统无法明确给出下一步，则本规范视为未成立。

---

## 12. v1 草案边界

### 12.1 v1 是什么

- 一个可跑通的施工企业项目经营闭环草案
- 一个可驱动后续实现收口的业务基线
- 一个后续可以继续修订到 v2/v3 的产品定义起点

### 12.2 v1 不是什么

- 不是完整 ERP 业务蓝图
- 不是财务核算系统规范
- 不是合同/税务/发票的完整体系
- 不是多行业共用顶层标准

---

## 13. 当前待讨论问题

本版已纳入当前确认的三个修正结论：

1. 业务阶段和发布切片必须分离
2. 成本和付款不应直接升格为项目主阶段
3. 只有阶段性动作才能推进 `project.stage`

后续继续修订时，优先围绕以下问题展开：

- 结算前置条件是否需要进一步细化为正式规则集合
- executing 阶段内各里程碑是否还需要增加或合并
- milestone 写回触发行为是否需要单独形成动作规范
- 驾驶舱如何基于 stage + milestone 稳定生成 next_actions

---

## 14. 总结

这份草案的本质，不是把系统描述成“能做若干功能”，而是把系统收口成“能完成一条施工企业项目业务链”。

后续所有实现与验收，应逐步向这条业务闭环靠拢，而不是继续堆叠孤立功能。
