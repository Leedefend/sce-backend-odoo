# 施工企业管理系统启用主路径总纲 v1

- 版本：`v1`
- 日期：`2026-03-26`
- 状态：`Frozen for Execution`

## 一、唯一总目标

把系统从“可开发、可验证”推进到“可启用、可运行、可进入项目执行主线”。

本阶段唯一正确顺序固定为：

```text
系统初始化
→ 组织与人员就位
→ 权限生效
→ 项目开始运行
→ 任务进入执行
```

## 二、阶段划分

### Phase 0：运行环境初始化

目标：
- 建立真实业务主体、组织关系、角色边界与可用账号。

必须先完成：
- 公司基本信息
- 组织架构
- 角色定义
- 用户设置

完成标准：
- 后续项目、成员、任务都能挂到“公司/部门/用户/角色”四要素上。

### Phase 1：项目运行基础闭环

目标：
- 让项目从创建进入最小执行闭环。

必须完成：
- 创建项目
- 配置项目负责人/归口部门
- 配置项目成员
- 创建任务
- 分派执行人
- 执行状态反馈

完成标准：
- 一个新建项目可以被真实用户接手、生成任务、推进状态并完成最小执行闭环。

## 三、本阶段不做

- 不做高阶风险治理扩展
- 不做高级驾驶舱优化
- 不做复杂审批链设计
- 不做多租户高级治理能力
- 不做超出最小可启用范围的主数据平台化重构

## 四、核心原则

### 原则 1：先能启用，再谈治理完美

- 公司、部门、角色、用户只做最小可用版。

### 原则 2：所有配置必须服务“项目能跑起来”

- 禁止为了配置而配置。

### 原则 3：前端路径必须围绕启用顺序组织

- 登录后应优先引导：
  `公司 → 部门 → 角色/用户 → 项目 → 任务`

### 原则 4：当前阶段暂不回到高阶经营能力扩展

- `risk / evidence / dashboard` 已具备基础闭环，但不是当前优先施工面。

### 原则 4.1：前端产品化采用双轨承接策略

- 冻结文档：[frontend_takeover_scope_v1.md](/mnt/wsl/docker-desktop-bind-mounts/Ubuntu/0bf88c91312832ece483d20f9dd0da58b3449c7beac0658c5397b284fcec1f13/docs/product/frontend_takeover_scope_v1.md)
- 当前阶段不再要求“所有对象先完整进入统一原生视图解析主链，才允许产品化”。
- 高频、标准、普通用户主业务链页面，允许由自定义前端直接承接。
- 低频、复杂、高耦合、系统治理类页面，保留 Odoo 原生兜底。
- 前端承接与原生兜底的边界，以冻结清单为唯一判断依据。

### 原则 5：每轮冲刺必须是用户可见的端到端切片

- Sprint 完成定义不再接受“只完成事实层 / 模型层 / guard 层”。
- 每轮必须同时做到：
  - 用户能进入入口
  - 用户能完成操作
  - 用户能看到结果
  - 用户不会迷路
  - verify 通过
- 任何只在底层生效、但用户无法感知的改动，都不能单独作为一个 Sprint 收口。

### 原则 6：每轮 Sprint 强制覆盖五层闭环

- 任何 Sprint 都必须同时包含以下五层，缺一层视为未完成：
  - 事实层：模型、字段、关系、约束
  - 契约层：接口、contract、scene/page/zone/block 或最小返回结构
  - 入口层：菜单、按钮、跳转、创建入口、详情入口
  - 交互层：保存、提示、刷新、错误态、空态、返回路径
  - 验证层：guard、smoke、snapshot、用户路径验证

### 原则 7：每轮采用双验收标准

- 工程验收：
  - upgrade 通过
  - verify 通过
  - 不破坏旧链路
- 用户验收：
  - 必须能由人完成 `进入 → 创建/修改 → 保存 → 查看结果 → 继续下一步`
- 只通过工程验收、不通过用户验收，视为该 Sprint 未完成。

### 原则 8：没有用户回路就不开工、不收口

- 任何需求在进入实现前都必须先回答四个问题：
  - 用户从哪里进入？
  - 用户点什么完成操作？
  - 操作成功后看见什么？
  - 如果失败，系统怎么提示？
- 任一问题答不出来，不进入实现。
- 没有用户可见入口，不开工。
- 没有用户操作回路，不收口。

### 原则 9：企业主数据与组织权限底座不放行业模块

- `res.company`
- `hr.department`
- `res.users`
- `res.groups / role matrix`

上述能力都属于企业启用底座，不属于施工行业域。

- 行业模块只能消费这些主数据，不能拥有它们的真源实现。
- `smart_construction_core` 只允许读取、关联、校验，不允许成为公司/组织/角色的主实现模块。
- 若当前仓库缺少合适承载模块，应先新建基础模块，再推进 Sprint 实现。

### 原则 10：前端承接必须先做分类，再决定实现路径

- 不再以“能不能解析原生视图”为唯一排期标准。
- 每个页面进入实现前，必须先被归类为：
  - 前端承接
  - 原生保留
  - 条件承接
- 凡是被归类为原生保留的页面，不进入当前阶段前端重做排期。
- 凡是被归类为条件承接的页面，必须先具备能力判断器和 `open_native` 降级路径，再逐步接管。

## 五、产品交付主线

### 主线 1：业务主体初始化

- 先建立公司，再让部门、用户、项目都归属到公司。

### 主线 2：责任关系初始化

- 先建立部门树，再明确部门负责人和上下级关系。

### 主线 3：权限可用性初始化

- 先冻结基础角色，再让不同用户真正拥有不同入口和操作能力。

### 主线 4：项目运行启动

- 在主体、组织、角色、用户都齐备后，再开放项目、成员和任务主链。

## 六、里程碑定义

### Milestone M0：系统可启用

达成条件：
- 完成 Phase 0
- 可以创建公司、部门、用户并赋予角色

### Milestone M1：项目可运行

达成条件：
- 完成 Phase 1
- 可以创建项目、配置成员、创建任务并完成状态推进

## 七、后续冲刺边界

本阶段后续所有迭代必须围绕以下主路径，不允许跳步：

```text
Sprint 0：公司 + 组织
Sprint 1：角色 + 用户
Sprint 2：项目创建 + 项目成员
Sprint 3：任务创建 + 分派 + 状态执行
Sprint 4：产品化启用主路径收口（引导、验收、门禁）
```

若任一前置 Sprint 未通过验收，禁止进入下一 Sprint。

## 七点五、产品交互迭代落位顺序

在不破坏 Sprint 顺序的前提下，后续产品交互迭代必须优先覆盖冻结清单中的前端主承接对象：

- 第一批：公司、组织基础、用户基础、项目、任务
- 第二批：付款申请基础页、成本记录基础页、合同基础页
- 第三批：工作台、项目驾驶舱、门户协同页

复杂结算、权限治理、高耦合系统页不进入当前阶段前端重绘主排期。

## 八、冲刺执行新原则 v1

- 每轮只做一个用户可感知主题。
- 每轮必须包含入口、页面、动作、反馈、验证。
- 不接受只落模型，不接受只落 verify。
- 没有用户路径演示，不算完成。
- 所有底层工作都必须绑定到当前 Sprint 的用户动作上。
