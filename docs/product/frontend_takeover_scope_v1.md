# 前端承接范围清单 v1

- 版本：`v1`
- 日期：`2026-03-26`
- 状态：`Frozen`
- 适用阶段：`当前产品交付阶段`
- 架构策略：`高频标准页前端承接，低频复杂页原生兜底`
- 冻结依据：`当前能力只能判定为 form 主链可用、list 主链可用、kanban 轻量可用，尚未达到任意原生 form/tree/kanban 全能力承载`

## 一、冻结原则

### 1. 前端承接目标

前端只承接三类高频标准页面：

- 标准列表页
- 标准表单页
- 轻量看板页

### 2. 原生保留目标

原生保留三类页面：

- 复杂交互页
- 高耦合事务页
- 系统治理与配置页

### 3. 完成标准

一个功能只有同时满足下面条件，才允许进入前端承接范围：

- 普通用户高频使用
- 交互模式标准化
- 当前 contract 和渲染器能稳定支撑
- 不依赖复杂原生行为
- 即使降级也不破坏业务主链

### 4. 总体边界

当前阶段不追求“全量替代 Odoo 原生界面”，只追求：

> **覆盖普通用户主业务链，复杂能力自动降级到原生。**

## 二、前端承接能力白名单

### A. 标准列表页

当前允许前端承接的列表能力：

- 字段列展示
- 列顺序
- 默认排序
- 基础搜索
- 基础筛选
- 分页
- 分组展示
- 行点击打开
- 多选
- 基础批量操作
- 导出当前结果
- 基础状态标签
- 基础工具栏按钮

说明：标准列表主链可承接，但搜索生态、工具栏、收藏夹、searchpanel 仍不算完整能力。

### B. 标准表单页

当前允许前端承接的表单能力：

- 基础字段渲染
- group 分组布局
- notebook 简单页签
- header 按钮
- button_box/stat button 基础承接
- 基础状态栏
- 基础附件/chatter 区块
- 简单 x2many 展示
- 新建/编辑/保存
- 基础按钮执行
- 基础只读/必填/显隐表达

说明：表单主链最强，但仍不是完整等价原生承接。

### C. 轻量看板页

当前允许前端承接的看板能力：

- 卡片标题
- 基础字段摘要
- 简单分组
- 基础 quick action
- 基础状态语义
- 轻量卡片列表展示

说明：当前只支持轻量卡片式看板，不可宣称完整原生看板承接。

## 三、前端禁止承接黑名单

以下能力在 v1 阶段一律不进入前端承接范围：

### 1. 表单复杂行为

- 复杂 attrs/modifiers 联动
- 复杂 onchange
- 全量 domain/context/options 原生等价解释
- 任意复杂 form arch 无损还原

### 2. 子表复杂编辑

- 复杂 one2many inline 编辑
- 嵌套子表联动
- 高事务性子记录编辑流程

### 3. 列表高级生态

- 完整 searchpanel
- favorites 完整生态
- 原生 tree 全量 inline 行为
- 全量 toolbar/batch/search 统一语义承接

### 4. 看板高级交互

- 拖拽
- 泳道/列级完整保真
- 复杂 badge/tag/progress
- 深度原生模板消费
- 复杂卡片行为反馈

### 5. 原生高耦合系统页

- 会计深度页面
- 采购/库存复杂事务页
- 复杂 wizard
- 高级审批配置页
- 权限治理配置页
- 系统参数治理页

这些能力当前一旦命中，统一走原生。

## 四、模块落位表 v1

下面按施工企业管理系统当前阶段，正式冻结各模块落位。

### 1. 公司管理

- 前端承接：`是`
- 原生保留：`是`

前端承接范围：
- 公司列表
- 公司基础表单
- 基础启停
- 基础查询

原生保留范围：
- 多公司复杂治理
- 深层公司配置
- Odoo 原生公司高级设置

结论：
> 公司信息作为标准 list/form，可前端承接；复杂公司治理保留原生。

### 2. 组织架构

- 前端承接：`部分承接`
- 原生保留：`是`

前端承接范围：
- 部门列表
- 部门基础表单
- 基础树形展示
- 负责人选择
- 所属公司归属

原生保留范围：
- 复杂组织调整
- 深层层级维护
- 与 HR 体系深耦合扩展

结论：
> 组织架构前端承接基础维护，复杂组织治理保留原生。

### 3. 用户管理

- 前端承接：`部分承接`
- 原生保留：`是`

前端承接范围：
- 用户列表
- 用户基础资料
- 公司/部门关联
- 基础状态启停

原生保留范围：
- 用户安全设置
- 高级访问控制
- 登录认证高级配置
- 多角色复杂治理

结论：
> 用户主数据可前端承接，安全与权限深层管理保留原生。

### 4. 角色与权限

- 前端承接：`否`
- 原生保留：`是`

原因：
- 高耦合 Odoo group / ACL / record rule
- 容易造成前后端权限真源漂移

结论：
> 角色与权限在 v1 一律原生，不做前端重绘。

### 5. 项目管理

- 暂未被前端稳定覆盖的复杂页签

## 五、统一降级规则

为了让这个清单能执行，正式冻结以下降级规则。

### 1. 降级触发条件

页面命中任一条件时，自动降级原生：

- 命中黑名单能力
- 前端能力判断器判定不支持
- 页面 contract 不完整
- 页面行为依赖复杂原生逻辑
- 当前渲染结果不能保证主链稳定

### 2. 降级动作

统一动作名：

`open_native`

### 3. 降级原则

- 降级优先于猜测渲染
- 降级优先于半成品交互
- 降级优先于前端临时拼逻辑

## 六、执行纪律

从 v1 起，后续所有页面实现必须遵守下面规则：

### 1

不再以“能不能解析”为唯一判断标准。

### 2

必须先判断页面属于：

- 前端承接
- 原生保留
- 条件承接

### 3

凡是被归类为原生保留的页面，不再进入前端重做排期。

### 4

凡是被归类为前端承接的页面，必须做到用户可见全链路闭环。

### 5

凡是条件承接页面，必须先实现能力判断器与降级逻辑，再逐步接管。

## 八、最终冻结结论

正式冻结一句话策略：

> **普通用户高频标准业务页由自定义前端承接；复杂、低频、高耦合页面由原生 Odoo 兜底。**

以及一句执行口径：

> **不是重写全部 Odoo 前端，而是覆盖业务主链，保留原生兜底。**
