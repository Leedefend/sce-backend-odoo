# SCBSLY 费用/扣款/保证金办理口径收口 v1

日期：2026-06-14

## 当前事实

`sc.expense.claim` 是费用、扣款、保证金和往来还款的统一承载模型，不是三张独立单据。业务事实需要同时看：

- `business_category_id`：正式业务分类入口。
- `business_axis`：业务概念，回答“这属于费用、扣款、保证金还是往来款”。
- `claim_type`：资金/事项大类。
- `expense_type`：旧系统事项文本和新系统默认事项。
- `direction`：资金方向。
- `handling_kind`：本次新增的用户可见办理口径。
- `financial_flow`：财务概念，回答“这是现金流入、现金流出、非现金事实还是内部往来资金”。
- `payment_anchor_policy`：本次新增的付款/收款申请锚点口径。

## 新旧系统差异

新系统办理规则仍然是：

- 费用报销、保证金支付/退回、扣款实缴/退回：正式新单必须关联付款/收款申请。
- 扣款登记：办理动作和列表入口；其下的扣款单明细表达公司与项目/承包人之间的代扣列支和责任清分事实，不关联付款/收款申请。
- 往来还款：作为内部往来款，不关联经营付款/收款申请。
- 历史迁移事实：允许不关联付款/收款申请，但通过办理口径保留业务事实。

关键区分：

- 扣款登记是动作/列表；扣款单是具体明细事实。扣款单的业务实质是公司先扣除税费、管理费等应扣事项，再在项目或承包人责任关系中列支清分；财务影响为非现金，不等同于付款或收款。
- 扣款实缴是财务现金流入，不应因为 `claim_type=expense` 被理解成费用流出。
- 扣款退回是财务现金流出，不应因为旧 `direction=inflow` 被理解成收款。
- 费用报销是业务费用，财务影响通常为现金流出。
- 保证金要区分缴纳/支付、退回、收取/自筹保证金；它们同属保证金业务，但财务方向不同。
- 承包人还项目款、项目还公司款等属于往来款业务，不能混入经营付款/收款申请链。

扣款登记、扣款单、扣款实缴需要分开理解：

- 扣款登记：用户办理动作和列表入口。
- 扣款单：扣款登记下的具体明细，确认“应扣并列支”的业务责任事实，减少可拨付/可结算责任余额，不直接形成银行收支。
- 扣款实缴：被扣税费等实际缴入公司或相关账户，是现金流入，需走收款申请锚点。
- 扣款退回：原已实缴扣款发生退回，是现金流出，需走付款申请锚点。

旧系统历史数据中，`deposit_receive` 有一批原本被业务分类归到“项目费用报销单”的记录。本次通过 `handling_kind` 将其拆清为“自筹保证金”或“保证金收取”，避免用户在费用入口误解保证金事实。

## 已修正

- 新增 `handling_kind`，用于列表、表单、搜索和分组展示真实办理口径。
- 新增 `business_axis`，用于展示系统性业务概念。
- 新增 `financial_flow`，用于展示系统性财务影响。
- 新增 `payment_anchor_policy`，用于展示申请锚点规则。
- 费用/保证金列表和表单展示上述口径字段。
- 提交、批准、完成时的付款/收款申请类型校验改为依据 `financial_flow`：
  - `cash_in` 必须关联收款申请。
  - `cash_out` 必须关联付款申请。
  - `noncash` 和 `interfund` 不应关联经营付款/收款申请。
- 现金流入/流出的账户必填校验改为依据 `financial_flow`，不再依赖旧 `direction`。
- 审计快照记录 `business_axis`、`handling_kind`、`financial_flow`、`payment_anchor_policy`。
- 普通报销分类名称修正为“报销申请”，项目费用分类保留“项目费用报销单”。
- `sc_demo` 已回算 51407 条历史单据口径，业务概念和财务概念均已覆盖。
